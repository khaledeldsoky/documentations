دليل GitHub Actions الشامل
من الصفر إلى CI/CD

GitHub Actions هو نظام CI/CD مدمج في GitHub يخلّيك تبني وتختبر وتنشر كودك تلقائيا . الدليل ده مصمم للمبتدئين اللي معندهمش أي خلفية عن CI/CD أو YAML . هنشرح كل مفهوم من الصفر مع أمثلة عملية .

CI/CD GitHub Actions YAML DevOps
1

ما هو GitHub Actions ؟

GitHub Actions هو نظام CI/CD ( Continuous Integration / Continuous Deployment ) مدمج في GitHub . ببساطة : أي حاجة بتعملها بإيدك بعد ما تعمل push — زي إنك تشغل tests أو تنشر الموقع — تقدر تخلي GitHub يعملها تلقائيا .

❌
قبل CI/CD — يدوي
تعمل push ← تفتح terminal ← تشغل tests ← تبني ← ترفع على السيرفر . يضيع وقت ، تنسى خطوة ، يحصل خطأ .
✅
بعد CI/CD — تلقائي
تعمل push ← GitHub يشغل tests ← يبني ← ينشر تلقائيا . أنت بس تركز على الكود .
معمارية GitHub Actions
Workflow ( الملف كله - .github/workflows/*.yml )
Job 1 (مثلا: build — بيشتغل على Ubuntu)
Step 1: actions/checkout (جيب الكود)
Step 2: actions/setup-node (ثبت Node)
Step 3: run: npm ci (نزل الحزم)
Step 4: run: npm run build (ابني)
Job 2 (مثلا: deploy — محتاج Job 1 يخلص)
Step 1: download artifact (جيب الملفات)
Step 2: deploy to server (انشر)
المفاهيم الأساسية
المصطلحالشرحمثال
Workflowملف YAML بيحدد الأتمتة كلها.github/workflows/deploy.yml
Jobمجموعة steps بتشتغل على نفس الـ runnerbuild , test , deploy
Stepأمر أو action واحد داخل jobrun: npm test
Actionوحدة reusable — بتيجي من GitHub Marketplaceactions/checkout@v4
Runnerالسيرفر اللي بيشتغل عليه الـ workflowubuntu-latest , self-hosted
Eventالحدث اللي بيشغل الـ workflowpush , pull_request , schedule
ℹ
خلينا نتفق على حاجة : الـ Workflow هو الملف كله . الـ Job هو مهمة معينة داخل الملف . الـ Step هو سطر واحد في المهمة . متحفظش الكلام ده — افهم العلاقة بينهم وهترسمها في دماغك .
2

أول Workflow لك

خلينا نبدأ بأصغر Workflow ممكن — يطبع رسالة لما تعمل push . مش محتاج غير ملف واحد في المكان الصح .

أين تنشئ الملف ؟
المسار الإجباري 
# أنشئ الفولدر ده في نفس مسار الـ repo بتاعك
mkdir -p .github/workflows/

# بعدين أنشئ ملف .github/workflows/first.yml
touch .github/workflows/first.yml
⚠
مهم جدا : الملف لازم يكون جوا .github/workflows/ بالظبط . لو حطيته في أي مكان تاني , GitHub مش هيعرف يقراه . الإمتداد يكون .yml أو .yaml .
أقل Workflow ممكن
first.yml — Hello World 
# اسم الـ Workflow (بيظهر في GitHub)
name: Hello World

# متى يشتغل؟
on:
  push:                      # أي push على أي branch

# المهام
jobs:
  hello:                     # اسم الـ job (أي اسم)
    runs-on: ubuntu-latest # السيرفر اللي هيشتغل عليه
    steps:
      - run: echo "Hello from GitHub Actions!"

بعد ما تعمل push للملف ده , روح لـ GitHub ← تبويب Actions ← هتشوف الـ Workflow شغال . دوس عليه وشوف الـ logs .

إزاي تشوف النتيجة ؟
  1. أعمل push للملف خلي الملف في .github/workflows/first.yml واعمل commit + push .
  2. افتح GitHub ( الـ repo بتاعك ) هتلاقي تبويب اسمه Actions في أعلى الصفحة — دوس عليه .
  3. شوف الـ Workflow بيشتغل هتلاقي " Hello World " في القائمة — دوس عليه وشوف الـ logs live .
  4. لما يطلع علامة خضرا ✅ — مبروك ! أول Workflow لك شغال ! طبع " Hello from GitHub Actions ! " في الـ log .
الخلاصة

Workflow = YAML file in .github/workflows/ , triggered by an Event , runs Jobs with Steps on a Runner .

3

بنية YAML — شرح كل كلمة

خلينا ناخد كل keyword في YAML ونفهمه لوحده . كل كلمة ليها دور معين في الـ workflow . مش محتاج تحفظ — افهم الدور وهتعرف تستخدمه .

on — متى يشتغل الـ Workflow ؟

دي أول كلمة في الـ workflow . بتحدد الحدث اللي يشغل الملف . ممكن يكون push ، pull_request ، schedule ( مؤقت ) ، أو يدوي ( workflow_dispatch ) .

أمثلة on 
# على كل push
on: push

# على push لـ main أو dev بس
on:
  push:
    branches: [main, dev]

# على pull_request لـ main
on:
  pull_request:
    branches: [main]

# جدول زمني — كل يوم 12 midnight UTC
on:
  schedule:
    - cron: '0 0 * * *'

# يدوي — زرار في GitHub
on:
  workflow_dispatch:
    inputs:
      env:
        description: 'Environment'
        required: true
        default: 'staging'
ℹ
معلومة : تقدر تجمع أكتر من event في نفس الـ workflow . مثلا يشتغل على push و pull_request في نفس الوقت .
jobs — المهام

كل workflow فيه job أو أكتر . كل job بتشتغل على runner منفصل ( سيرفر مستقل ) . انت بتسمي الـ job أي اسم يناسبك — من غير مسافات .

jobs — basic 
jobs:
  build:           # ⬅️ أي اسم تختاره
    runs-on: ubuntu-latest
    steps: [ ... ]

  deploy:          # ⬅️ job تاني
    runs-on: ubuntu-latest
    steps: [ ... ]
الخاصيةالشرحمثال
runs-onنوع الـ runnerubuntu-latest , windows-latest
needsالـ job اللي لازم يخلص الأولneeds: build
ifشرط التشغيلif: github.ref == 'refs/heads/main'
strategyتكرار الـ job بقيم مختلفةmatrix
envمتغيرات للـ job كلهenv: NODE_ENV: test
permissionsصلاحيات الـ jobpermissions: contents: read
steps — الخطوات

كل job فيه steps بتتنفذ بالترتيب ( واحدة ورا التانية ) . لو step فشلت , الباقي مش هيتنفذ ( إلا لو حطيت continue-on-error: true ) .

أنواع الـ steps 
steps:
  # use — استخدم action جاهز من GitHub
  - uses: actions/checkout@v4

  # run — أمر shell مباشر
  - name: Install dependencies
    run: npm ci

  # with — parameters للـ action
  - uses: actions/setup-node@v4
    with:
      node-version: '20'

  # run مع multi-line
  - name: Build and test
    run: |
      npm run build
      npm test

  # if — شرط للـ step
  - if: failure()
    run: echo "Something failed!"
runs-on — على إيه يشتغل الـ Job ؟

كل job محتاج سيرفر يشتغل عليه . أنت بتختار نوع السيرفر ده . GitHub-hosted بتوع GitHub ، و Self-hosted بتوعك أنت .

runs-on — options 
jobs:
  linux:
    runs-on: ubuntu-latest   # أو ubuntu-22.04

  windows:
    runs-on: windows-latest # أو windows-2022

  mac:
    runs-on: macos-latest   # أو macos-14

  private:
    runs-on: self-hosted    # على جهازك أنت
needs — ترتيب الـ Jobs

بشكل افتراضي , كل الـ jobs بتشتغل بالتوازي ( كل job على runner لوحدها في نفس الوقت ) . needs بيخلي job تستنى job تانية تخلص الأول .

needs — dependencies 
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building..."

  test:
    needs: build            # ⬅️ يستنى build يخلص
    runs-on: ubuntu-latest
    steps:
      - run: echo "Testing..."

  deploy:
    needs: [build, test] # ⬅️ يستنى build و test
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying..."
💡
شكل التدفق : build ← test ← deploy . لو build فشل , test مش هيتشغل أصلا .
strategy.matrix — تشغيل متعدد

الـ matrix يخلّيك تكرر نفس الـ job بقيم مختلفة ( مثلا تختبر على 3 أنظمة تشغيل و 3 نسخ Node في نفس الوقت ) .

Matrix — OS × Node 
jobs:
  test:
    strategy:
      fail-fast: false    # لو واحد فشل، الباقي يكمل
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]

    runs-on: ${{ matrix.os }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci && npm test

الـ code ده هينتج 9 runs ( 3 OS × 3 Node versions ) — كلهم بيشتغلوا بالتوازي . تقدر تضيف include أو exclude عشان تتحكم في الـ combinations .

env — المتغيرات

تقدر تعرّف متغيرات على 3 مستويات : للـ workflow كله ( global ) ، للـ job ، أو للـ step واحد .

env — 3 levels 
# Global — لكل الـ workflow
env:
  APP_NAME: my-app
  NODE_ENV: production

jobs:
  build:
    # Job level — للـ job ده بس
    env:
      BUILD_DIR: ./dist

    steps:
      - name: Build
        # Step level — للـ step ده بس
        env:
          API_URL: https://api.example.com
        run: echo $APP_NAME $BUILD_DIR $API_URL
permissions — الصلاحيات

بتحدد صلاحيات الـ GITHUB_TOKEN ( التوكن اللي GitHub بيديه للـ workflow ) . أحسن ممارسة ( best practice ) إنك تقلل الصلاحيات قد ما تقدر .

permissions — examples 
# أقل صلاحية — قراءة فقط (الأفضل)
permissions:
  contents: read

# للـ job واحد بس
jobs:
  deploy:
    permissions:
      contents: read
      pages: write      # عشان النشر على GitHub Pages
      id-token: write
if — الشروط

بتستخدم if عشان تتحكم في job أو step — يشتغل بس لو الشرط تحقق . تقدر تستخدم متغيرات github ( زي github.ref ) أو دوال زي failure() و always() .

if — conditions 
# Job condition — بس على main
jobs:
  deploy:
    if: github.ref == 'refs/heads/main'

    steps:
      # Step condition
      - if: failure()       # بس لو حصل فشل
        run: echo "Failed!"

      - if: always()        # دايما — حتى لو فشل
        run: echo "Cleanup..."

      - if: success()       # لو نجح — دا السلوك الافتراضي
        run: echo "All good!"
المتغيرالقيمة مثالالشرح
github.refrefs/heads/mainالـ branch أو tag الحالي
github.event_namepush , pull_requestالحدث اللي شغّل الـ workflow
github.actorkhaledاسم اليوزر اللي عمل الحدث
github.repositoryowner/repoالـ repo الحالي
github.shaabc123...الـ commit hash
defaults — الإعدادات الافتراضية

بتحدد إعدادات افتراضية لكل الـ run steps — زي shell أو working directory . بدل ما تكرر نفس الإعدادات في كل step .

defaults 
defaults:
  run:
    shell: bash
    working-directory: ./app

jobs:
  build:
    steps:
      - run: pwd  # هيكون في ./app
      - run: ls   # هيكون في ./app
ملخص سريع لكل YAML
Full Workflow Template 
name: Workflow Name     # اسم الـ workflow

on: [push]                # متى يشتغل؟

env:                      # متغيرات عامة
  KEY: value

jobs:                     # المهام
  job-name:               # اسم الـ job (أي اسم)
    runs-on: ubuntu-latest # نوع الـ runner
    needs: [job1]           # dependencies
    if: github.ref == '...' # شرط
    permissions: {}          # صلاحيات

    steps:                   # الخطوات
      - uses: actions/checkout@v4
      - name: Description
        run: command
        env:                    # متغيرات للـ step
          KEY: value
        if: condition
4

من النظري للعملي

خلينا نطبق اللي اتعلمناه على سيناريو حقيقي : موقع React ( أو Vue أو أي SPA ) عايزين نبنيه وننشره على GitHub Pages — كل ما تعمل push على main .

السيناريو — 7 خطوات
📖
القصة : أنت developer عندك موقع React . بتكتب كود , تعمل push , وعايز الموقع ينشر تلقائيا من غير ما تفتح terminal أو تدوس أي زرار .
  1. أنت بتعمل : git push origin main GitHub يستقبل الـ push ويعرف إن فيه event حصل على main .
  2. GitHub يدور على ملفات الـ Workflow بيروح لـ .github/workflows/ ويلاقي ملف deploy.yml — يبدا يقراه .
  3. GitHub يوفّر Runner بيجهز سيرفر ubuntu-latest عليه Git و Node.js وجاهز للشغل .
  4. Job 1 : Build Checkout ← Setup Node ← npm ci ← npm run build → ينتج مجلد build/ .
  5. Upload Artifact يحفظ مجلد build/ عشان الـ job الجاي يستخدمه .
  6. Job 2 : Deploy ( needs : build ) يستنى build يخلص ← يحمل artifact ← ينشر على GitHub Pages .
  7. الموقع LIVE ! 🚀 الموقع الجديد شغال على https://user.github.io/repo/ .
الرسمة الكاملة
Job 1: BUILD
runs-on: ubuntu-latest
├─ actions/checkout@v4 ← "جبت الكود ✅"
├─ actions/setup-node@v4 ← "ثبت Node.js 20 ✅"
├─ run: npm ci ← "نزلت dependencies ✅"
├─ run: npm run build ← "بنيت → build/ ✅"
└─ upload-artifact ← "حفظت build/ ✅"
↓ needs : build " مستنى build يخلص . . . " ↓
Job 2: DEPLOY
needs: build
if: github.ref == 'refs/heads/main'
├─ download-artifact ← "جبت build/ ✅"
└─ deploy-pages ← "نشرت على Pages ✅"
← "الموقع LIVE! 🚀"
الـ YAML الكامل
deploy.yml — React إلى GitHub Pages 
name: Build & Deploy React App

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: production-files
          path: ./build/

  deploy:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: production-files
          path: ./build/
      - uses: actions/deploy-pages@v4
        id: deployment
مقارنة — نظري vs عملي
النظريالعملي ( YAML )
" يشتغل على push لـ main " on: push: branches: [main]
" على Ubuntu " runs-on: ubuntu-latest
" جيب الكود " uses: actions/checkout@v4
" ثبت Node " uses: actions/setup-node@v4
" نزل الحزم " run: npm ci
" ابني الموقع " run: npm run build
" احفظ الملفات " uses: actions/upload-artifact@v4
" مستنى build يخلص " needs: build
" جيب الملفات " uses: actions/download-artifact@v4
" انشر " uses: actions/deploy-pages@v4
الخلاصة

You don't need to memorize YAML syntax . Understand the flow : Event → Job → Step . The syntax is just a way to describe that flow in a file .

5

GitHub-hosted vs Self-hosted

الـ Runner هو السيرفر اللي بيشتغل عليه الـ workflow . نوعين : GitHub-hosted ( بيوفره GitHub ) و Self-hosted ( بتوفره أنت على جهازك أو سيرفرك ) .

الخاصيةGitHub-hostedSelf-hosted
السيرفرسيرفرات GitHubجهازك أو VM أو سيرفرك
التكلفة2000 دقيقة/شهر مجانا ( Public repos ) مجاني — لكن تكلفة السيرفر عليك
الإعدادجاهز — مش محتاج حاجةتحتاج تنزل وتشغل GitHub Actions Runner
الوصول للشبكةعام — مش هيوصل لـ private networkيوصل لشبكتك الداخلية — VPN , VPC , إلخ
الصيانةGitHub مسئولعليك — أمان , تحديثات , توفر
الاستخدام99% من الحالات — يكفيلما تحتاج hardware خاص أو private network
إعداد Self-hosted Runner
تثبيت Runner على Ubuntu 
# 1. روح للـ repo → Settings → Actions → Runners → Add runner
# 2. GitHub هيديك أوامر زي دي:

mkdir actions-runner && cd actions-runner
curl -o actions-runner-linux-x64-2.xxx.tar.gz -L https://github.com/actions/runner/releases/download/v2.xxx/actions-runner-linux-x64-2.xxx.tar.gz
tar xzf actions-runner-linux-x64-2.xxx.tar.gz
./config.sh --url https://github.com/YOUR-ORG/YOUR-REPO --token ABC123
./run.sh
استخدام الـ self-hosted في YAML 
jobs:
  deploy:
    runs-on: self-hosted   # هيشتغل على جهازك
    steps:
      - run: echo "Running on my own server!"
⚠
تنبيه أمان : الـ self-hosted runner أي شخص عنده access للـ repo يقدر يشغل كود على جهازك . استخدمه بحذر — خصوصا في public repos . الأفضل تستخدم GitHub-hosted لو تقدر .
6

ربط ملفات YAML

مع增大 المشروع , هتحتاج تعيد استخدام نفس الـ steps في أكتر من workflow . GitHub Actions يدعم طريقتين للربط :

الطريقة 1 : Reusable Workflows

Workflow عادي لكنه بيستخدم workflow_call بدل push — وده يخلّي workflows تانية تستدعيه .

reusable-build.yml — الـ Workflow القابل لإعادة الاستخدام 
name: Reusable Build

on:
  workflow_call:             # ⬅️ دي اللي تخليه reusable
    inputs:
      node-version:
        required: true
        type: string

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm ci && npm run build
main.yml — الاستدعاء 
jobs:
  call-reusable:
    uses: ./.github/workflows/reusable-build.yml
    with:
      node-version: '20'
الطريقة 2 : Composite Actions

Action صغير reusable — بيجمع مجموعة steps في ملف action.yml . الفرق إنه مش workflow كامل , ده مجرد steps جاهزة تستخدمها جوا أي job .

. github/actions/my-setup/action.yml 
name: My Setup
description: Setup Node and install deps

runs:
  using: composite
  steps:
    - uses: actions/setup-node@v4
      with:
        node-version: '20'
    - run: npm ci
      shell: bash
استخدام الـ composite action 
jobs:
  build:
    steps:
      - uses: actions/checkout@v4
      - uses: ./.github/actions/my-setup  # ⬅️ استدعاء
      - run: npm run build
💡
الفرق : Reusable workflows = workflow كامل , Composite actions = steps جاهزة للاستخدام داخل أي job .
مثال — 3 ملفات مرتبطة
.github/
├── workflows/
│ ├── main.yml # الـ workflow الرئيسي
│ └── reusable-test.yml # reusable workflow
└── actions/
└── setup-node/ # composite action
└── action.yml
main.yml — يستدعي reusable-test و composite action 
name: Main CI/CD
on: [push]

jobs:
  build:
    steps:
      - uses: actions/checkout@v4
      - uses: ./.github/actions/setup-node
      - run: npm run build

  call-tests:
    needs: build
    uses: ./.github/workflows/reusable-test.yml

  deploy:
    needs: [build, call-tests]
    steps:
      - run: echo "Deploying..."

الأخطاء الشائعة & Debugging

أكتر الأخطاء اللي بتقع فيها وأنت بتتعلم GitHub Actions — وكيف تحلها .

المشكلةالسببالحل
الـ workflow مش بيظهر في GitHubالملف مش في المسار الصحتأكد من .github/workflows/*.yml
" Invalid workflow file " YAML indentation غلطاستخدم مسافتين ( مش tabs )
" Command not found " نسيت actions/checkoutحط - uses: actions/checkout@v4 أول step

Context Objects — المتغيرات المضمنة

GitHub Actions فيه Context Objects — متغيرات جاهزة تقدر تستخدمها في الـ YAML عشان تجيب معلومات عن الـ repo , الـ event , الـ runner , وغيرها . بتستخدمها بصيغة ${{ context.key }} .

المتغيرالوصفأمثلة
${{ github.ref }}الـ branch أو tag الحاليrefs/heads/main
${{ github.event_name }}الحدث اللي شغّل الـ workflowpush , pull_request
${{ github.actor }}اليوزر اللي عمل الحدثkhaled
${{ github.repository }}الـ repo الحاليowner/repo-name
${{ github.sha }}الـ commit hashabc123def456...
${{ env.VAR_NAME }}متغير من env${{ env.NODE_ENV }}
${{ secrets.SECRET_NAME }}secret من Settings${{ secrets.API_KEY }}
${{ matrix.os }}قيمة من الـ matrixubuntu-latest
${{ runner.os }}نظام الـ runnerLinux , Windows , macOS
أمثلة استخدام Context 
jobs:
  info:
    steps:
      - run: echo "Branch: ${{ github.ref }}"
      - run: echo "Event: ${{ github.event_name }}"
      - run: echo "Actor: ${{ github.actor }}"
      - run: echo "OS: ${{ runner.os }}"
      - env:
          MY_SECRET: ${{ secrets.API_KEY }}
        run: echo "Key length: ${#MY_SECRET}"
⚠
تحذير أمان : متطبعش الـ secrets في الـ logs ! لو كتبت echo ${{ secrets.API_KEY }} , GitHub بيحولها لـ *** تلقائيا . بس لسه — بلاش تطبعها .

الأمان في GitHub Actions

🔐
1 . أقل صلاحية مطلوبة
دايمًا استخدم permissions : contents : read — متديش أكتر من اللي الـ job محتاجه .
🔑
2 . OIDC بدل Secrets
OIDC بيخلّي الـ workflow ياخد token مؤقت من مزود السحابة ( AWS , Azure , GCP ) من غير ما تخزن secret طويل العمر .
🛡️
3 . تجنب Script Injection
متستخدمش ${{ }} جوا run: مباشرة — ممكن user يدخل كود خبيث عن طريق PR title مثلا . استخدم env: عشان تمرر القيم .
📌
4 . Pin Actions بـ SHA
بدل @v4 , استخدم @full-sha-hash عشان تمنع التغييرات المفاجئة في الـ action من تأثير على الـ workflow .
Script Injection — المشكلة والحل 
# ❌ خطأ — قابل للاختراق
- run: echo "${{ github.event.pull_request.title }}"

# ✅ صح — آمن
- env:
    PR_TITLE: ${{ github.event.pull_request.title }}
  run: echo "$PR_TITLE"
ℹ
خلاصة الأمان : أقل صلاحية→ OIDC إن أمكن→ لا تثق في input من الخارج→ حدد version الـ actions .

أسئلة المقابلات

أكثر الأسئلة اللي بتتكرر في مقابلات GitHub Actions — مع الإجابات المثالية .

أسئلة أساسية
Q
اشرح GitHub Actions باختصار .
•هو نظام CI/CD من GitHub . بتكتب ملفات YAML في .github/workflows/ تحدد فيها Events ( زي push ) اللي تشغل Jobs ( مهام ) فيها Steps ( خطوات ) على Runners ( سيرفرات ) .
Q
إيه الفرق بين push و pull_request events ؟
•push بيشتغل لما تعمل push على branch . pull_request بيشتغل لما تعمل PR ( open , synchronize , closed ) .
Q
إيه الفرق بين needs و strategy.matrix ؟
•needs بيخلّي job تستنى job تانية ( تنفيذ متسلسل ) . •matrix يخلّيك تكرر نفس الـ job بقيم مختلفة ( تنفيذ متوازي ) .
Q
إزاي تعمل deployment بعد ما tests تعدي ؟
•استخدم needs: test في job الـ deploy — مش هيتشغل غير لو test نجح . وكمان if: github.ref == 'refs/heads/main' عشان بس على main .
سيناريوهات عملية في المقابلات
Q
لو عندك workflow بيشتغل على push لـ main , وعايز تعمل manual trigger باختيار البيئة .
•استخدم workflow_dispatch مع input للـ environment .
Manual Trigger 
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment to deploy'
        required: true
        default: 'staging'
Q
ازاي تسرع الـ workflow بواسطة caching ؟
•استخدم actions/cache@v4 عشان تحفظ node_modules أو ~/.npm وتستعيدها في المرات الجاية بدل ما تنزل من الصفر .
Caching node_modules 
- name: Cache node_modules
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-node-
🎯
نصيحة للمقابلة : ابدأ بالـ basics ( Workflow ← Job ← Step ) . اذكر actions/checkout و actions/setup-node كأمثلة . تكلم عن caching و secrets كـ best practices . لو سألوك عن scale : اذكر reusable workflows و self-hosted runners .
