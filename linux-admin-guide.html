عملية الإقلاع — Linux Boot Process

بدل ما تحفظ الموضوع كخطوات صماء ، حاول تتخيله كقصة بتحصل من أول ما تدوس زرار الـ Power . كل خطوة بتجهز اللي بعدها ، والـ Boot Process هو المسلسل اللي بيحصل في كل مرة تشغل فيها السيرفر .

التسلسل الكامل من Power On إلى Login
  1. Power On : أول ما تشغل السيرفر ، الـ CPU بيبدأ يدور على أول كود يتنفذ — بيروح للـ Firmware ( BIOS أو UEFI ) .
  2. BIOS / UEFI — POST يفحص الهاردوير : يتأكد إن الرام شغالة ، البروسيسور شغال ، والهارد موجود . بعد كده يدور على Boot Device حسب ترتيب الـ Boot Order ( USB , SSD , DVD ) .
  3. تحميل الـ Bootloader ( GRUB ) الـ BIOS/UEFI مش بيعرف يشغل Linux Kernel مباشرة . فبيحمل GRUB — المدير بتاع الإقلاع . وظيفته : يعرض قائمة الأنظمة ( لو عندك Dual Boot ) ، يختار Kernel ، ويمرر Parameters .
  4. GRUB يحمل Kernel GRUB يحمل ملفين مهمين : /boot/vmlinuz ( Linux Kernel ) و /boot/initramfs.img ( initramfs ) .
  5. Kernel يبدأ الشغل الـ Kernel هو قلب النظام . أول ما يشتغل : يدير الرام ، يدير الـ CPU ، ويحمل Drivers . بس فيه مشكلة — الـ Root Filesystem لسه مش متوصل .
  6. initramfs دي Mini Linux صغيرة جدًا . وظيفتها : تحميل Drivers ، اكتشاف الأقراص ، تفعيل LVM ، وفك تشفير الديسك لو مشفر . بعد ما تعرف مكان النظام الحقيقي ، تعمل mount / وترمي نفسها .
  7. Mount Root Filesystem يعمل mount / للنظام الحقيقي ( مثلاً /dev/mapper/rhel-root ) ويرمي الـ initramfs .
  8. تشغيل PID 1 — systemd أول Process في النظام . في الأنظمة الحديثة هو systemd . رقمه دائمًا PID = 1 .
  9. systemd يشغل الخدمات يشتغل Network , SSH , Cron , Docker , وكل الخدمات المطلوبة .
  10. Login Screen في النهاية : Login Prompt ( سيرفر ) أو GUI ( جهاز مكتبي ) .

3 . /bin vs /sbin — الفرق

زمان كان فيه فرق واضح . لكن في الأنظمة الحديثة بقت مجرد Symlinks لـ /usr/bin و /usr/sbin .

المجلدزمانالنهاردة
/binأوامر أساسية لكل المستخدمين ( ls , cp , mv ) /usr/bin
/sbinأوامر إدارة النظام ( reboot , fdisk ) /usr/sbin
/bin و /sbin في الأنظمة الحديثة 
# لو عملت ls -ld على /bin و /sbin
ls -ld /bin /sbin
# lrwxrwxrwx 1 root root 7 ... /bin -> usr/bin
# lrwxrwxrwx 1 root root 8 ... /sbin -> usr/sbin
ℹ
خلاصة : كلهم بقوا في /usr/bin و /usr/sbin . الفرق بقا في الاستخدام : /usr/bin لأوامر كل المستخدمين ، /usr/sbin لأوامر الإدارة ( تحتاج sudo غالباً ) .
4 . إضافة Scripts خاصة بيك — /usr/local/bin

عاوز تضيف Command جديد للنظام ؟ أحط Script في /usr/local/bin واديله صلاحية تنفيذ . المكان ده مخصص للبرامج والمشاريع المحلية .

إضافة Script بايثون كـ Command 
# إنشاء ملف Script
sudo vi /usr/local/bin/hello

# المحتوى:
#!/usr/bin/env python3
print("Hello from Python")

# صلاحية التنفيذ
sudo chmod +x /usr/local/bin/hello

# استخدمه من أي مكان
hello
# Hello from Python
✓
ليه /usr/local/bin ؟
• لأن /bin و /usr/bin تبع النظام نفسه . لو حطيت حاجتك في /usr/local/bin ، مش هتتلخبط مع Updates النظام .
5 . الـ PATH و Copy vs Symlink

لما تكتب أمر في الـ Terminal ، النظام يدور عليه في الأماكن اللي في متغير اسمه $PATH .

PATH — النظام بيجيب الأمر منين ؟ 
# شوف الـ PATH
echo $PATH
# /usr/local/bin:/usr/bin:/usr/sbin

# اعرف مكان الأمر
which ls
# /usr/bin/ls

# اعرف مكان أمر من /sbin
which reboot
# /usr/sbin/reboot

# زود مسار في الـ PATH (يختفي بعد logout)
export PATH=$PATH:/opt/myapp/bin

# عشان يبقى دايم — ضيف السطر ده في ~/.bashrc
echo 'export PATH=$PATH:/opt/myapp/bin' >> ~/.bashrc
Copy vs Symlink — ليه النقل مش فكرة كويسة ؟

ممكن تقنيًا تعمل cp /sbin/ip /bin/ip ، لكن ده غلط . الأفضل تعمل Symlink :

Symlink — أحسن من Copy 
# Copy (مشكوك فيه — نسختين من البرنامج)
sudo cp /usr/sbin/ip /usr/bin/ip
# المشكلة: Update هيحدث واحدة والتانية لا → لخبطة

# Symlink (صح — ملف واحد بس)
sudo ln -s /usr/sbin/ip /usr/local/bin/ip

# أو زود /usr/sbin في PATH
echo 'export PATH=$PATH:/usr/sbin' >> ~/.bashrc
⚠
سؤال إنترفيو مشهور : " هل تقدر تنقل برنامج من /sbin لـ /bin ؟ "
• الجواب : Technically yes , but it ' s not recommended . Use PATH or a symbolic link instead to avoid maintenance and update issues .
6 . خلاصة الإنترفيو
إجابة السؤال : " احكيلي عن Linux Filesystem "

Linux filesystem starts from the root directory / . Important directories are /boot for boot files , /etc for configuration files , /home for user data , /root for the root user , /var for logs , /tmp for temporary files , /usr for applications and binaries , /dev for devices , and /proc for kernel and process information . Linux treats almost everything as a file — regular files , directories , devices , and even processes are represented as files in the filesystem tree .

Hard Link vs Symbolic Link

الروابط في لينكس تسمح بإنشاء مراجع متعددة لنفس الملف . هناك نوعان : Hard Link و Symbolic Link ( Soft Link ) . الفرق بينهما جوهري .

ℹ
معلومة أساسية : كل ملف في لينكس له inode — رقم فريد يخزن بيانات الملف ( metadata ) ما عدا الاسم . اسم الملف هو مجرد رابط ( link ) لرقم inode معين .
جدول المقارنة
الخاصيةHard LinkSymbolic Link ( Soft Link )
الأمر ln source target ln -s source target
رقم inode نفس inode المصدر inode مختلف ( ملف جديد )
حذف المصدر الرابط يظل شغال — الملف لم يُحذف فعليًا الرابط ينكسر ( broken link )
عبر الأقسام لا — لا يمكن عبر filesystems مختلفة نعم — يمكن عبر أي مسار
للمجلدات لا — لا يمكن إنشاء hard link لمجلد نعم — يمكن إنشاء رابط لمجلد
المسار النسبي/المطلق غير مهم المسار مهم — إذا تحرك الملف ينكسر الرابط
أمثلة عملية
Hard Link — مثال 
# إنشاء ملف أصلي
echo "Hello World" > original.txt

# إنشاء Hard Link
ln original.txt hardlink.txt

# نفس inode — هما نفس الملف
ls -li original.txt hardlink.txt
# 123456 -rw-r--r-- 2 user user 12 ... original.txt
# 123456 -rw-r--r-- 2 user user 12 ... hardlink.txt

# حذف الأصلي — الرابط لسه شغال
rm original.txt
cat hardlink.txt  # Hello World ✓
Symbolic Link — مثال 
# إنشاء ملف أصلي
echo "Hello World" > original.txt

# إنشاء Soft Link
ln -s original.txt softlink.txt

# inodes مختلفة — ls -l يظهر الرابط
ls -l softlink.txt
# lrwxrwxrwx 1 user user 12 ... softlink.txt -> original.txt

# حذف الأصلي — الرابط ينكسر
rm original.txt
cat softlink.txt  # No such file or directory ✗
✓
متى تستخدم كل نوع ؟
• استخدم Hard Links لعمل نسخ احتياطية ذكية ( مثل rsnapshot ) حيث لا تستهلك مساحة إضافية . استخدم Symbolic Links للاختصارات والروابط عبر الأقسام وللمجلدات .

SSH — تأمين الاتصال عن بُعد

SSH ( Secure Shell ) هو البروتوكول الأكثر استخدامًا لإدارة الخوادم عن بُعد . تأمين إعدادات SSH هو أول خطوة بعد تثبيت أي خادم لينكس .

قفل المستخدم root ( منع تسجيل الدخول المباشر )

تسجيل الدخول مباشرة بحساب root عبر SSH يعتبر ممارسة غير آمنة . الأفضل تسجيل الدخول بحساب عادي ثم استخدام sudo أو su .

/etc/ssh/sshd_config — منع root 
# قم بتعديل هذا السطر في ملف /etc/ssh/sshd_config
# منع تسجيل الدخول المباشر للمستخدم root
PermitRootLogin no

# أو يمكنك استخدام 'prohibit-password' للسماح بمفاتيح SSH فقط
PermitRootLogin prohibit-password
تحقق من التعديل

grep " ^PermitRootLogin " /etc/ssh/sshd_config

تغيير منفذ SSH ( الـ Port )

تغيير المنفذ الافتراضي ( 22 ) يقلل من هجمات المسح الآلي . اختر منفذًا عاليًا ( 1024–65535 ) .

تغيير البورت في sshd_config 
# في ملف /etc/ssh/sshd_config
Port 2222

# يمكنك تحديد أكثر من منفذ
Port 2222
Port 443
⚠
تحذير : قبل إعادة تحميل SSH ، تأكد أن المنفذ الجديد مفتوح في الـ Firewall . وإلا ستغلق على نفسك !
استخدام المفاتيح ( SSH Keys )

المفاتيح أكثر أمانًا من كلمات المرور . استخدم ssh-keygen لإنشاء مفتاح ، ثم انسخه للخادم .

إنشاء مفتاح SSH ونسخه 
# إنشاء مفتاح (على جهازك المحلي)
ssh-keygen -t ed25519 -C "user@host"

# نسخ المفتاح للخادم
ssh-copy-id -p 2222 user@server-ip

# أو يدويًا: أضف المحتوى لـ ~/.ssh/authorized_keys
cat ~/.ssh/id_ed25519.pub
# ثم الصق المحتوى في ~/.ssh/authorized_keys على الخادم
ℹ
ملاحظة : بعد التعديلات ، أعد تحميل الخدمة بـ systemctl reload sshd . الـ reload أفضل من restart لأنه لا يقطع الاتصالات الحالية .

Permissions — إدارة الصلاحيات في لينكس

الصلاحيات في لينكس تتحكم في مين يقدر يقرأ ، يكتب ، أو ينفذ ملف . نظام الصلاحيات من أقدم وأهم المفاهيم في لينكس .

chmod — تغيير الصلاحيات

طريقتين : رمزي ( symbolic ) ورقمي ( numeric ) .

chmod — الطريقة الرمزية 
# إضافة صلاحية تنفيذ للمالك
chmod u+x script.sh

# إزالة صلاحية الكتابة للمجموعة
chmod g-w file.txt

# إضافة قراءة للجميع
chmod +r file.txt

# كل الصلاحيات للمالك، قراءة/تنفيذ للباقي
chmod u=rwx,g=rx,o=rx script.sh
chmod — الطريقة الرقمية 
# rwx = 7, rw- = 6, r-x = 5, r-- = 4
chmod 755 script.sh    # rwxr-xr-x
chmod 644 file.txt     # rw-r--r--
chmod 700 private.sh   # rwx------
chmod 600 secret.txt   # rw-------
chmod 777 public.sh    # rwxrwxrwx — خطير!
⚠
تحذير : chmod 777 يخلي أي مستخدم يعدل وينفذ الملف . استخدمه بحذر شديد — غالبًا في خطأ .
chown , chgrp — تغيير المالك والمجموعة
تغيير المالك والمجموعة 
# تغيير المالك
chown ahmed file.txt

# تغيير المالك والمجموعة
chown ahmed:developers file.txt

# تغيير المجموعة فقط
chgrp developers file.txt

# مجلد وكل محتوياته
chown -R ahmed:developers /opt/myapp
umask — الصلاحيات الافتراضية

الـ umask يحدد الصلاحيات الافتراضية للملفات والمجلدات الجديدة .

Reload vs Restart — الفرق

بعد تعديل قواعد الـ firewall ، تحتاج لتطبيق التغييرات . هناك طريقتان :

الأمرالسلوكمتى تستخدم
firewall-cmd --reload يعيد تحميل القواعد بدون قطع الاتصالات القائمة ( όχι stateful ) التعديلات اليومية — لا يقطع الخدمات
firewall-cmd --complete-reload يقطع كل الاتصالات ويعيد بناء الـ rules من الصفر عند الحاجة لتطبيق تغييرات جذرية ( نادر )
systemctl reload firewalld مشابه لـ --reload لكن عن طريق systemd عند تعديل ملفات الـ config مباشرة
systemctl restart firewalld يوقف ويبدأ الخدمة — يقطع كل الاتصالات مؤقتًا إذا كانت الخدمة عالقة أو لا تستجيب
⚠
تذكير مهم : إذا أضفت قاعدة بـ --permanent فقط ولم تعمل --reload ، القاعدة لن تُفعل حتى الـ reload . العكس : إذا أضفت قاعدة بدون --permanent ، القاعدة نشطة فورًا ولكنها تختفي بعد الـ reload .

Network Ports — تحديد المنافذ والاتصالات

لو عايز تعرف مين ماسك Port معين على لينكس وجاي منين ( IP/Process ) ، عندك مجموعة أوامر قوية . الفرق بين Listening ( المنفذ مفتوح ) و Established ( فيه اتصال فعلي ) هو أول حاجة تفهمها .

1 . تعرف مين فاتح الـ Port — Listening

الأفضل والأحدث استخدام ss بديل netstat .

ss -lntp | grep : PORT 
# مين فاتح port 80؟
ss -lntp | grep :80
# LISTEN 0  511  *:80   *:*   users:(("nginx",pid=1234,fd=6))

# شرح:
# LISTEN     → المنفذ مفتوح ويستقبل اتصالات
# nginx      → اسم البرنامج
# pid=1234   → رقم الـ Process
# *:80       → بيستمع على كل الـ IPs على port 80
ℹ
معنى ال flags : -l ( listening فقط ) ، -n ( أرقام — لا يحل أسماء ) , -t ( TCP فقط ) , -p ( يظهر الـ process ) .
2 . تشوف كل الاتصالات الفعلية — Established

الفرق من اللي فوق : من غير -l عشان تظهر الاتصالات اللي حاصلة فعلاً مش اللي مستنية .

ss -ntp | grep : PORT 
# مين متصل على port 80 دلوقتي؟
ss -ntp | grep :80
# ESTAB  0  0  192.168.1.50:54000  192.168.1.10:80  users:(("nginx",pid=1234))

# شرح:
# ESTAB           → اتصال قائم فعلاً
# 192.168.1.50    → IP العميل اللي متصل
# 54000           → Port العميل (عشوائي)
# 192.168.1.10    → السيرفر بتاعك
# 80              → Port السيرفر
3 . طريقة قوية جداً — lsof

أمر lsof يعطيك تفاصيل كاملة عن الملفات المفتوحة ومنها الـ network connections .

lsof -i : PORT 
# تفاصيل كاملة عن port 443

# COMMAND  PID   USER   FD   TYPE  DEVICE  NODE  NAME
# nginx    1234  root   6u   IPv4  12345   TCP   *:https (LISTEN)
# nginx    1235  www    7u   IPv4  12346   TCP   10.0.0.1:443->10.0.0.2:56000 (ESTABLISHED)

# يظهر لك: الـ Process، PID، المستخدم، نوع الاتصال، الـ IP المحلي والبعيد
⚠
ملاحظة : lsof محتاج صلاحيات sudo عشان يشوف كل العمليات . بديله لو مش متوفر : ss -tulnp ( بـ sudo ) .

XFS vs Ext4 — مقارنة أنظمة الملفات

XFS و Ext4 هما أشهر أنظمة الملفات في لينكس . اختيار نظام الملفات المناسب يعتمد على حجم البيانات ، نوع الـ workload ، ومتطلبات الاستقرار .

نظرة عامة
🚀
XFS — للملفات الكبيرة والأداء العالي
نظام ملفات 64-bit عالي الأداء . ممتاز للملفات الكبيرة جدًا ( multi-GB/TB ) . يدعم حتى 8 EB كحد أقصى . مناسب لأنظمة التخزين الضخمة و databases . يعمل online defragmentation ( xfs_fsr ) . نظام RHEL default .
💪
Ext4 — الموثوقية والتوافق
خليفة Ext3 ، الأكثر اختبارًا في الأجهزة المختلفة . يدعم حتى 1 EB . سهل الإصلاح ( fsck ) . مناسب للملفات المتوسطة والكثيرة ( الكثير من الملفات الصغيرة ) . يستخدم extents لتقليل التجزئة . نظام Ubuntu default .
الخاصيةXFSExt4
أقصى حجم للملف8 EB16 TB
أقصى حجم لنظام الملفات8 EB1 EB
الأداء مع الملفات الكبيرةممتاز ( +1 GB ) جيد
الأداء مع الملفات الصغيرةمتوسطممتاز
Grow ( تكبير ) نعم — بدون unmountنعم
Shrink ( تصغير ) لانعم
Online Defragنعم ( xfs_fsr ) لا ( e4defrag تجريبي )
نظام RHEL default✔️❌
نظام Ubuntu default❌✔️
أدوات الصيانة — Repair & Check

xfs_fsr /dev/sda1   # defrag

# Ext4
fsck.ext4 -y /dev/sda1
tune2fs -l /dev/sda1 | grep -i "filesystem state"

2 . التقسيم ( Partitioning )

الطريقةالحد الأقصىنظام التقسيممتى تستخدم

3 . MBR vs GPT — مقارنة شاملة

اختيار نظام التقسيم يعتمد على حجم الديسك ونظام الإقلاع ( BIOS vs UEFI ) .

الخاصيةMBR ( BIOS ) GPT ( UEFI )

التوافق مع Windows✔️ قديم✔️ حديث
الاستخدامأقراص صغيرة / أنظمة قديمةجميع الأقراص الحديثة ( +2 TB )

# أو بـ gdisk

# n → New partition
# w → Write & exit
⚠
تذكر : تغيير MBR إلى GPT بيمسح كل البيانات على الديسك . اعمل Backup أولاً .
4 . LVM — الـ Pipeline الكامل

LVM هو طريقة التخزين الأشهر في السيرفرات . بدل ما تستخدم الديسك مباشرة ، بتمر بطبقات :

LVM Pipeline : PV → VG → LV 
Disk / Partition
    ↓
Physical Volume (PV)    ← pvcreate /dev/sdb
    ↓
Volume Group (VG)        ← vgcreate data_vg /dev/sdb
    ↓
Logical Volume (LV)      ← lvcreate -L 20G -n app_lv data_vg
    ↓
Filesystem               ← mkfs.xfs /dev/data_vg/app_lv
    ↓
Mount Point              ← mount /dev/data_vg/app_lv /data
مثال كامل — LVM من الصفر 
# 1. Physical Volume
pvcreate /dev/sdb
pvs                           # تأكيد

# 2. Volume Group
vgcreate data_vg /dev/sdb
vgs                           # تأكيد

# 3. Logical Volume (20G)
lvcreate -L 20G -n app_lv data_vg
lvs                           # تأكيد

# 4. Filesystem
mkfs.xfs /dev/data_vg/app_lv

# 5. Mount
mkdir /data
mount /dev/data_vg/app_lv /data
df -h                         # تأكيد
ℹ
تشبيه بسيط : PV = فلوس في المحافظ ، VG = حساب بنكي جمعت فيه الفلوس ، LV = المبالغ اللي بتوزعها على المشاريع . عشان كده LVM قوي — تقدر تجمع ديسكات مختلفة في VG واحدة وتوزع المساحة بالطريقة اللي تعجبك .
5 . إنشاء Filesystem

بعد ما خلصت من LVM أو الـ Partition ، لازم تعمل Filesystem عشان تقدر تستخدم المساحة . ( راجع Section 13 للمقارنة الكاملة بين XFS و Ext4 ) .

mkfs — إنشاء نظام ملفات 
# XFS (الأفضل للسيرفرات — RHEL default)
mkfs.xfs /dev/data_vg/app_lv

# Ext4 (متوافق — يمكن تصغيره)
mkfs.ext4 /dev/data_vg/app_lv

# أو على Partition مباشر
mkfs.xfs /dev/sdb1
⚠
تذكر : XFS يقبل الـ Grow ( تكبير ) فقط — لا يمكن تصغيره . Ext4 يقبل Grow و Shrink . راجع Section 13 للتفاصيل .
6 . التركيب ( Mount ) و /etc/fstab

عشان يستخدم المستخدمين المساحة ، لازم تركبها على مجلد معين .

Mount و fstab 
# إنشاء مجلد للـ mount
mkdir /data

# Mount مباشر
mount /dev/data_vg/app_lv /data

# تأكيد

# Filesystem               Size  Used Avail Use% Mounted on
# /dev/mapper/data_vg-app_lv  20G  1G   19G   5% /data

# ── للـ persistence بعد الـ reboot ──

# اعرف الـ UUID

# /dev/data_vg/app_lv: UUID="123-456" TYPE="xfs"

# ضيف السطر ده في /etc/fstab
# UUID=123-456  /data  xfs  defaults  0  0

# اختبر قبل الـ reboot

التأكد النهائي

df -h | grep /data

# /dev/mapper/data_vg-app_lv 20G 1G 19G 5% /data

7 . توسيع المساحة ( LVM Extension )

الميزة الحقيقية لـ LVM — تقدر تكبر المساحة بدون إعادة تقسيم أو إيقاف الخدمات .

lvextend — تكبير LV 
# تكبير LV +10G — الر_flag -r تكبر الـ FS تلقائياً
lvextend -r -L +10G /dev/data_vg/app_lv

# أو استخدم كل المساحة الفاضية
lvextend -r -l +100%FREE /dev/data_vg/app_lv
8 . دمج عدة أقراص في VG واحدة

أقوى ميزة في LVM — تقدر تجمع أقراص متعددة الأحجام في VG واحدة .

دمج أقراص متعددة 
# عندك: sdb = 100G, sdc = 100G, sdd = 200G

# حولهم كلهم لـ PV
pvcreate /dev/sdb /dev/sdc /dev/sdd

# VG واحدة تجمعهم
vgcreate vg_data /dev/sdb /dev/sdc /dev/sdd

# النتيجة: VG = 400G
vgs
# vg_data  3 PVs  400G

# بعد سنة — لو جالك ديسك جديد
pvcreate /dev/sde
vgextend vg_data /dev/sde
# VG = 500G بدون ما تلمس الـ LVs أو الـ Applications
✓
الصورة النهائية لـ LVM في دماغك : PV → VG → LV → FS → Mount . ودي السلسلة اللي لو فهمتها ، 80% من أسئلة LVM هتبقى سهلة .

cron لجدولة مهام متكررة ، at لتنفيذ مهمة مرة واحدة في وقت محدد . كلاهما أدوات أساسية لأي مدير نظام .

Cron — المهام المتكررة

ملفات cron موجودة في عدة أماكن :

الملف/المجلدالوصف
/etc/crontabالملف الرئيسي — فيه حقل user إضافي
/etc/cron.d/مجلد لملفات cron إضافية ( نفس صيغة crontab )

/etc/cron.daily/سكريبتات تنفذ يوميًا
/etc/cron.weekly/سكريبتات تنفذ أسبوعيًا
/etc/cron.monthly/سكريبتات تنفذ شهريًا
crontab -eكرون خاص بالمستخدم الحالي
صيغة الـ Crontab 
# ┌───────────── minute (0–59)
# │ ┌───────────── hour (0–23)
# │ │ ┌───────────── day of month (1–31)
# │ │ │ ┌───────────── month (1–12)
# │ │ │ │ ┌───────────── day of week (0–7, 0 = Sunday)
# * * * * * command

# أمثلة:

# كل يوم الساعة 2:30 صباحًا

# أول يوم من كل شهر الساعة 12 ظهرًا

0 18 * * 5 /usr/bin/weekly-cleanup.sh
ℹ
تذكير : بعد تعديل الـ crontab ، مش محتاج restart لأي خدمة — cron بيقرأ الملفات مباشرة . استخدم crontab -l لعرض المهام الحالية . وتأكد إن crond شغال بـ systemctl status crond .
At — التنفيذ لمرة واحدة

لو عاوز سكريبت يتنفذ مرة واحدة بس في وقت محدد ( مثلاً : تحديث في الساعة 3 الفجر ) ، استخدم at .

أمثلة لأمر at 
# تنفيذ أمر الساعة 10 مساءًا اليوم
echo "systemctl restart nginx" | at 22:00

# أو باستخدام التفاعل
at 22:00
at> systemctl restart nginx
at> # اضغط Ctrl + D

# أمثلة لصيغ الوقت:
at now + 5 minutes     # بعد 5 دقائق
at 14:30 tomorrow       # بكرة الساعة 2:30
at 3:00 AM                # الساعة 3 صباحًا
at 10:00 next week        # الأسبوع الجاي الساعة 10
at midnight               # نص الليل
at teatime                   # 4:00 PM

# عرض المهام المجدولة
atq
# إلغاء مهمة
atrm 2   # 2 = job number من atq

Nginx vs Apache — مقارنة بين خوادم الويب

Nginx و Apache هما أشهر خوادم الويب في العالم . كل واحد له فلسفته ونقاط قوته . اختيار المناسب يعتمد على حالة الاستخدام .

المقارنة الشاملة
⚡
Nginx
Event-driven architecture — يعالج آلاف الاتصالات المتزامنة بخيط واحد ( thread ) . أداء عالي جدًا في الثابت والمحتوى الثابت . استهلاك ذاكرة منخفض . ممتاز كـ reverse proxy و load balancer .
🔧
Apache
Process-based architecture — كل اتصال يأخذ thread أو process . مرونة عالية مع .htaccess والتعديل الديناميكي لكل دليل . عدد ضخم من الـ modules ( mod_rewrite , mod_ssl , etc . ) .
الخاصيةNginxApache
المعماريةEvent-driven ( async ) Process/Thread-based
المحتوى الثابتممتاز — سريع جدًاجيد
المحتوى الديناميكييحتاج FastCGI ( PHP-FPM ) مدمج ( mod_php )
. htaccessيدعم بس في الـ config الرئيسييدعم في أي مجلد — مرونة عالية
استهلاك الذاكرةقليل — مناسب للـ VPSأعلى — يعتمد على عدد الـ modules المحملة
الأداء في الـ High Concurrencyممتاز — آلاف الاتصالاتيمكن أن يضعف مع الاتصالات الكثيرة
Reverse Proxyمدمج وممتازيدعم عبر mod_proxy
صعوبة التعلممتوسط — config بسيط ومركزسهل للمبتدئين
أنظمة التشغيلUnix-like ( Windows limited ) جميع الأنظمة ( Windows native )
مقارنة الـ Config 
# Nginx — ملف config واحد لل server
server {
    listen       80;
    server_name  example.com;
    root         /var/www/html;
    location / {
        try_files $uri $uri/ /index.html;
    }
}

# Apache — يعتمد على Directory blocks و .htaccess
<VirtualHost *:80>
    DocumentRoot /var/www/html
    ServerName   example.com
    <Directory /var/www/html>
        AllowOverride All
    </Directory>
</VirtualHost>
ℹ
الخلاصة : استخدم Nginx لو عاوز performance عالي ، reverse proxy ، static files . استخدم Apache لو محتاج .htaccess وتوافق مع scripts قديمة أو لو فريقك معتاد عليه . كثير من الناس يستخدمون Nginx كـ reverse proxy قدام Apache ( الأفضل من الاثنين ) .

# إيقاف واستئناف

SIGKILL vs SIGTERM : دايمًا استخدم kill ( بدون رقم ) — SIGTERM — الأول . SIGTERM يسمح للعملية بتنظيف الموارد ( close files , release memory ) . استخدم SIGKILL kill -9 فقط لو العملية مش مستجيبة .

Troubleshooting — تشخيص وحل المشاكل

Troubleshooting هو مهارة " البحث المنهجي " عن سبب المشكلة . مش مجرد حفظ أوامر — دي منهجية : عرّف المشكلة ، اجمع معلومات ، حلّل ، نفّذ الحل ، اختبر .

منهجية التشخيص — 5 خطوات
  1. تعريف المشكلة إيه اللي حصل بالضبط ؟ مستخدم قال " الموقع وقع " ولا فيه alert من monitoring ؟
  2. جمع معلومات logs ، status ، metrics ( dmesg ، journalctl ، top , df -h , ping ) . ابدأ من الواسع للضيق .
  3. تحليل إيه الـ pattern ؟ إيه اللي اتغير قبل المشكلة ؟ ( آخر تحديث ، config change ، زيادة traffic )
  4. اختبار الفرضية طبق حل صغير . لو مشكلة permissions ؟ اختبر بعملية وحدة . لو DNS ؟ جرب nslookup منサーバ .
  5. تأكيد الحل وتوثيقه المشكلة اتحلت ؟ اتأكد إنها مش هترجع تاني . وثق الخطوات .
مشاكل وحلول شائعة — Common Scenarios

HTTP 400 vs 500 — أخطاء HTTP

HTTP status codes طريقة السيرفر يقول للـ Client " إيه اللي حصل " . نقسمهم لقسمين كبيرين : 4xx ( خطأ من العميل ) و 5xx ( خطأ من السيرفر ) .

4xx — Client Errors ( خطأ من المستخدم )
🔍
400 Bad Request
الـ request مش مفهوم للسيرفر — syntax غلط ، header كبير ، payload مش مطابق للمواصفات .
🚫
401 Unauthorized
مش مخوّل — محتاج تسجيل دخول . Authentication مفقودة أو غير صحيحة .
⛔
403 Forbidden
السيرفر فهم الـ request بس مش عاوز يسمح — permissions ، IP block ، SELinux .
❓
404 Not Found
الـ resource مش موجود . أشهر خطأ — المسار غلط ، الملف محذوف ، link مكسور .
⏱
408 Request Timeout
العميل أخذ وقت طويل في إرسال الـ request . غالبًا بسبب internet بطيء أو proxy مشكلة .
🔄
429 Too Many Requests
Rate limiting — العميل جاوز الحد المسموح من الـ requests . شائع في APIs .
5xx — Server Errors ( خطأ من السيرفر )
💥
500 Internal Server Error
خطأ عام من السيرفر — ما فيش تفاصيل . غالبًا بسبب application crash ، PHP fatal error ، أو misconfiguration .
⚠
502 Bad Gateway
السيرفر ( كمه — Nginx ) حاول يتصل بـ upstream ( مثلاً PHP-FPM ) وما استجابش . أشهر مشكلة مع reverse proxy .
🔌
503 Service Unavailable
السيرفر مش قادر يخدم — overload ، maintenance ، أو connections exhausted .
⌛
504 Gateway Timeout
الـ upstream ما ردش في الوقت المحدد . غالبًا بسبب application بطيء أو database query تقيلة .

Rescue Mode & GRUB — إنقاذ النظام

في بعض الأحيان النظام مش عاوز يقلع — GRUB اتكسر ، الـ kernel مش عاوز يشتغل ، أو نسيت كلمة السر . Rescue Mode و GRUB Recovery هما أدواتك لإنقاذ الموقف .

Rescue Mode / Single User Mode

وضع الإنقاذ ( Rescue Mode ) بيدخلك على shell كـ root بدون ما تشغل الخدمات كلها — عشان تصلح مشاكل boot .

الدخول لوضع الإنقاذ — من GRUB 
# أثناء boot: اضغط Shift (BIOS) أو Esc (UEFI) عشان تظهر GRUB
#
# 1. Highlight أول kernel option
# 2. اضغط 'e' عشان تعديل boot entry
# 3. دور على السطر اللي يبدأ بـ linux  أو linux16
# 4. زود الكلمة دي في نهاية السطر:
init=/bin/bash

# أو لو systemd:
systemd.unit=rescue.target

# 5. اضغط Ctrl + X أو F10 للبوت
# 6. هتفتح shell كـ root — صلح اللي عاوزه

# بعد الإصلاح — إعادة تشغيل
exec /sbin/init
# أو
reboot -f
⚠
ملاحظة : لو استخدمت init=/bin/bash ، النظام بيهت mount كـ read-only . عشان تكتب تعديلات : mount -o remount,rw /
GRUB Recovery — إصلاح الـ Bootloader

لو GRUB نفسه اتكسر أو اتخرب ( مثلاً بعد تثبيت Windows ) ، لازم تعيد تثبيته .

إصلاح GRUB — من Live CD/USB 
# 1. Boot من Live CD/USB
# 2. افتح terminal

# 3. اعرف اسم الـ partition (مثلاً /dev/sda1 = /boot, /dev/sda2 = /)
lsblk

# 4. Mount جذر النظام
mount /dev/sda2 /mnt
mount /dev/sda1 /mnt/boot   # لو /boot في partition منفصل

# 5. Chroot — ندخل جوه النظام
for dir in /dev /proc /sys /run; do mount --bind $dir /mnt$dir; done
chroot /mnt

# 6. إعادة تثبيت GRUB
grub2-install /dev/sda         # أو grub-install حسب التوزيعة

# 7. إعادة بناء GRUB config
grub2-mkconfig -o /boot/grub2/grub.cfg   # CentOS/RHEL/Fedora
update-grub                                 # Ubuntu/Debian

# 8. خروج وإعادة تشغيل
exit
reboot
ℹ
الفرق : grub-install يكتب GRUB على MBR أو UEFI partition . grub-mkconfig ( أو grub2-mkconfig ) يبني ملف الـ config من الـ kernels المثبتة في /boot .
إعادة تعيين كلمة سر root
نسيت كلمة سر root — إعادة تعيينها 
# نفس خطوات Rescue Mode:
# 1. من GRUB اضغط 'e' على kernel entry
# 2. زود init=/bin/bash في نهاية سطر linux
# 3. Ctrl + X للبوت

# إعادة mount كـ read-write
mount -o remount,rw /

# تغيير كلمة سر root
passwd

# لو SELinux مفعل
touch /.autorelabel

# إعادة تشغيل
exec /sbin/init
