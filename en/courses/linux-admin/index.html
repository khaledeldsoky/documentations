Linux Boot Process

Instead of memorizing the topic as rigid steps , try to imagine it as a story from the moment you press the Power button . Each step prepares the next one , and the Boot Process is the series that happens every time you turn on the server .

Complete Sequence from Power On to Login
  1. Power On : When you first turn on the server , the CPU starts looking for the first code to execute — it goes to the Firmware ( BIOS or UEFI ) .
  2. BIOS / UEFI — POST Checks the hardware : ensures RAM is working , the processor is working , and the disk is present . Then it searches for the Boot Device according to the Boot Order ( USB , SSD , DVD ) .
  3. Loading the Bootloader ( GRUB ) BIOS/UEFI does not know how to directly boot the Linux Kernel . So it loads GRUB — the boot manager . Its job : shows the list of operating systems ( if you have Dual Boot ) , selects the Kernel , and passes Parameters .
  4. GRUB loads the Kernel GRUB loads two important files : /boot/vmlinuz ( Linux Kernel ) and /boot/initramfs.img ( initramfs ) .
  5. Kernel starts working The Kernel is the heart of the system . When it first starts : it manages RAM , manages the CPU , and loads Drivers . But there is a problem — the Root Filesystem is not yet connected .
  6. initramfs This is a very small Mini Linux . Its job : load Drivers , discover disks , activate LVM , and decrypt the disk if encrypted . Once it finds the real system location , it does mount / and discards itself .
  7. Mount Root Filesystem It does mount / to the real system ( e.g. /dev/mapper/rhel-root ) and discards initramfs .
  8. Starting PID 1 — systemd The first Process in the system . In modern systems it is systemd . Its number is always PID = 1 .
  9. systemd starts services Starts Network , SSH , Cron , Docker , and all required services .
  10. Login Screen Finally : Login Prompt ( Server ) or GUI ( Desktop ) .

If someone asks you in an interview : " Tell me about the Linux Filesystem " — don't memorize many names , understand the concept . Linux considers everything a File : regular file , folder , hard disk , partition , USB , Printer , Process info . Everything is handled through a single file tree starting from / .

4 . Adding Your Own Scripts — /usr/local/bin

Want to add a new Command to the system ? Put a Script in /usr/local/bin and give it execute permission . This location is for local programs and projects .

Add a Python Script as a Command 
# Create a Script file
sudo vi /usr/local/bin/hello

# Content:
#!/usr/bin/env python3
print("Hello from Python")

# Execute permission
sudo chmod +x /usr/local/bin/hello

# Use it from anywhere
hello
# Hello from Python
✓
Why /usr/local/bin ?
• Because /bin and /usr/bin belong to the system itself . If you put your stuff in /usr/local/bin , it won't get mixed up with system Updates .
5 . PATH and Copy vs Symlink

When you type a command in the Terminal , the system searches for it in the locations listed in a variable called $PATH .

PATH — Where does the system find commands ? 
# Check PATH
echo $PATH
# /usr/local/bin:/usr/bin:/usr/sbin

# Find command location
which ls
# /usr/bin/ls

# Find location of a command from /sbin
which reboot
# /usr/sbin/reboot

# Add a path to PATH (temporary, lost after logout)
export PATH=$PATH:/opt/myapp/bin

# To make it permanent — add this line to ~/.bashrc
echo 'export PATH=$PATH:/opt/myapp/bin' >> ~/.bashrc
Copy vs Symlink — Why copying is not a good idea ?

Technically you can do cp /sbin/ip /bin/ip , but this is wrong . It's better to create a Symlink :

Symlink — Better than Copy 
# Copy (dubious — two copies of the program)
sudo cp /usr/sbin/ip /usr/bin/ip
# Problem: Update will affect one copy but not the other → confusion

# Symlink (correct — single file only)
sudo ln -s /usr/sbin/ip /usr/local/bin/ip

# Or add /usr/sbin to PATH
echo 'export PATH=$PATH:/usr/sbin' >> ~/.bashrc
⚠
Famous interview question : " Can you move a program from /sbin to /bin ? "
• The answer : Technically yes , but it ' s not recommended . Use PATH or a symbolic link instead to avoid maintenance and update issues .
6 . Interview Summary
Answer to the question : " Tell me about the Linux Filesystem "

Linux filesystem starts from the root directory / . Important directories are /boot for boot files , /etc for configuration files , /home for user data , /root for the root user , /var for logs , /tmp for temporary files , /usr for applications and binaries , /dev for devices , and /proc for kernel and process information . Linux treats almost everything as a file — regular files , directories , devices , and even processes are represented as files in the filesystem tree .

Hard Link vs Symbolic Link

Links in Linux allow creating multiple references to the same file . There are two types : Hard Link and Symbolic Link ( Soft Link ) . The difference between them is fundamental .

ℹ
Basic info : Every file in Linux has an inode — a unique number that stores the file's data ( metadata ) except the name . The file name is just a link ( link ) to a specific inode number .
Comparison Table
FeatureHard LinkSymbolic Link ( Soft Link )
Command ln source target ln -s source target
Inode number Same inode as source Different inode ( new file )
Delete source Link still works — file not actually deleted Link breaks ( broken link )
Across partitions No — cannot span different filesystems Yes — can span any path
For directories No — cannot create a hard link to a directory Yes — can create a link to a directory
Relative/absolute path Not important Path is important — if the file moves, the link breaks
Practical Examples
Hard Link — Example 
# Create original file
echo "Hello World" > original.txt

# Create Hard Link
ln original.txt hardlink.txt

# Same inode — they are the same file
ls -li original.txt hardlink.txt
# 123456 -rw-r--r-- 2 user user 12 ... original.txt
# 123456 -rw-r--r-- 2 user user 12 ... hardlink.txt

# Delete original — link still works
rm original.txt
cat hardlink.txt  # Hello World ✓
Symbolic Link — Example 
# Create original file
echo "Hello World" > original.txt

# Create Soft Link
ln -s original.txt softlink.txt

# Different inodes — ls -l shows the link
ls -l softlink.txt
# lrwxrwxrwx 1 user user 12 ... softlink.txt -> original.txt

# Delete original — link breaks
rm original.txt
cat softlink.txt  # No such file or directory ✗
✓
When to use each type ?
• Use Hard Links for smart backups ( like rsnapshot ) as they don't consume extra space . Use Symbolic Links for shortcuts, cross-partition links, and directories .

SSH — Securing Remote Connections

SSH ( Secure Shell ) is the most widely used protocol for remote server management . Securing SSH settings is the first step after installing any Linux server .

Locking the root user ( Preventing direct login )

Logging in directly with the root account via SSH is considered an insecure practice . It is better to log in with a regular account and then use sudo or su .

/etc/ssh/sshd_config — Deny root 
# Modify this line in /etc/ssh/sshd_config
# Prevent direct root login
PermitRootLogin no

# Or use 'prohibit-password' to allow SSH keys only
PermitRootLogin prohibit-password
Verify the change

grep " ^PermitRootLogin " /etc/ssh/sshd_config

Changing the SSH Port

Changing the default port ( 22 ) reduces automated scanning attacks . Choose a high port ( 1024–65535 ) .

Change the port in sshd_config 
# In file /etc/ssh/sshd_config
Port 2222

# You can specify multiple ports
Port 2222
Port 443
⚠
Warning : Before reloading SSH , make sure the new port is open in the Firewall . Otherwise you will lock yourself out !
Using SSH Keys

Keys are more secure than passwords . Use ssh-keygen to create a key , then copy it to the server .

Create and copy an SSH key 
# Create a key (on your local machine)
ssh-keygen -t ed25519 -C "user@host"

# Copy the key to the server
ssh-copy-id -p 2222 user@server-ip

# Or manually: add the content to ~/.ssh/authorized_keys
cat ~/.ssh/id_ed25519.pub
# Then paste the content into ~/.ssh/authorized_keys on the server
ℹ
Note : After changes , reload the service with systemctl reload sshd . reload is better than restart because it doesn't disconnect current sessions .

Permissions — Managing Permissions in Linux

Permissions in Linux control who can read , write , or execute a file . The permissions system is one of the oldest and most important concepts in Linux .

chmod — Changing Permissions

Two methods : symbolic and numeric .

chmod — Symbolic Method 
# Add execute permission for owner
chmod u+x script.sh

# Remove write permission for group
chmod g-w file.txt

# Add read for everyone
chmod +r file.txt

# All permissions for owner, read/execute for others
chmod u=rwx,g=rx,o=rx script.sh
chmod — Numeric Method 
# rwx = 7, rw- = 6, r-x = 5, r-- = 4
chmod 755 script.sh    # rwxr-xr-x
chmod 644 file.txt     # rw-r--r--
chmod 700 private.sh   # rwx------
chmod 600 secret.txt   # rw-------
chmod 777 public.sh    # rwxrwxrwx — Dangerous!
⚠
Warning : chmod 777 lets any user modify and execute the file . Use it with extreme caution — it is usually a mistake .
chown , chgrp — Changing Owner and Group
Change owner and group 
# Change owner
chown ahmed file.txt

# Change owner and group
chown ahmed:developers file.txt

# Change group only
chgrp developers file.txt

# Directory and all its contents
chown -R ahmed:developers /opt/myapp
umask — Default Permissions

umask specifies the default permissions for new files and directories .

umaskFileDirectory
022644 ( rw-r--r-- ) 755 ( rwxr-xr-x )
002664 ( rw-rw-r-- ) 775 ( rwxrwxr-x )
077600 ( rw------- ) 700 ( rwx------ )
umask — Examples 
# Show current umask
umask
# 0022

# Change umask (for current session)
umask 077

# Permanent change — add to ~/.bashrc
echo 'umask 002' >> ~/.bashrc
SUID , SGID , Sticky Bit
PermissionSymbolMeaningExample
SUIDrwsr-xr-xExecute with owner's privileges/usr/bin/passwd
SGIDrwxrwsr-xExecute with group's privileges/usr/bin/write
Sticky BitrwxrwxrwtPrevents unauthorized deletion/tmp
Setting SUID , SGID , Sticky Bit 
# SUID — s replaces owner's x
chmod u+s /usr/bin/myapp

# SGID — s replaces group's x
chmod g+s /shared/dir

# Sticky Bit — t replaces others' x
chmod +t /shared/dir

# Or by number (4=SUID, 2=SGID, 1=Sticky)
chmod 4755 script.sh  # rwsr-xr-x
chmod 2755 shared.sh  # rwxr-sr-x
chmod 1777 /tmp       # rwxrwxrwt
⚠
Security : SUID is a security risk if used incorrectly . Programs with SUID like passwd and sudo — need it to interact with system files .
ACL — Access Control Lists

Regular permissions give you 3 levels ( owner , group , others ) . ACL allows finer-grained permissions for multiple users and groups .

getfacl and setfacl 
# Display ACL
getfacl file.txt
# # file: file.txt
# # owner: ahmed
# # group: developers
# user::rw-
# group::r--
# other::r--

# Add permission for a specific user
setfacl -m u:khaled:rwx file.txt

# Add permission for a specific group
setfacl -m g:devops:rx file.txt

# Remove ACL
setfacl -x u:khaled file.txt

# ACL for directory — inherited by new files
setfacl -m d:u:khaled:rwx /shared/project
ℹ
Note : Having an ACL on a file shows a + sign at the end of ls -l .

SELinux — Security Enhanced Linux

SELinux is an additional security system in Linux . Regular permissions ( DAC ) control who can do what . SELinux ( MAC ) adds a security layer — even if root has permissions , SELinux can block him .

Modes — SELinux Modes
ModeBehaviorWhen to use
EnforcingEnforces rules and blocks violationsProduction — full security
PermissiveLogs violations but doesn't blockDiagnostics — see what violations exist
DisabledSELinux completely offRarely — not secure
Check SELinux mode 
# Show current mode
getenforce
# Enforcing

# Full details
sestatus
# SELinux status:                 enabled
# Current mode:                   enforcing
# Mode from config file:          enforcing

# Change mode (temporary — lost after reboot)
setenforce 0    # Permissive
setenforce 1    # Enforcing

# Permanent change — edit /etc/selinux/config
vim /etc/selinux/config
# SELINUX=enforcing / permissive / disabled
⚠
RHCSA : If asked to disable SELinux , use setenforce 0 and edit /etc/selinux/config to permissive . Only use disabled if specifically told — because it requires a reboot .
SELinux Contexts — ls -Z , ps auxZ

Every file and process in SELinux has a Context . The Context is a tag that determines permissions .

View SELinux Contexts 
# File Context
ls -Z /var/www/html
# system_u:object_r:httpd_sys_content_t:s0 index.html

# Process Context
ps auxZ | grep httpd
# system_u:system_r:httpd_t:s0   apache  1234  ...

# User Context
id -Z
# unconfined_u:unconfined_r:unconfined_t:s0-s0:c0.c1023

Context format : user:role:type:level

Modifying Context — semanage , restorecon , chcon
SELinux Context modification tools 
# Change context (manual — not persistent)
chcon -t httpd_sys_content_t /var/www/html/index.html

# Reset context according to default rules
restorecon -v /var/www/html/index.html

# Full directory
restorecon -Rv /var/www/html

# Permanent context change — semanage
semanage fcontext -a -t httpd_sys_content_t "/custom/web(/.*)?"
restorecon -Rv /custom/web

# View fcontext rules
semanage fcontext -l | grep httpd
ℹ
Rule : Use restorecon first — if it doesn't work , use semanage fcontext for a permanent solution . chcon is temporary — lost after relabel .

Network Ports — Identifying Ports and Connections

If you want to know who is holding a specific Port on Linux and where it's coming from ( IP/Process ) , you have a set of powerful commands . The difference between Listening ( port is open ) and Established ( there is an actual connection ) is the first thing to understand .

1 . Find out who opened the Port — Listening

The best and most modern is to use ss instead of netstat .

ss -lntp | grep : PORT 
# Who opened port 80?
ss -lntp | grep :80
# LISTEN 0  511  *:80   *:*   users:(("nginx",pid=1234,fd=6))

# Explanation:
# LISTEN     → Port is open and accepting connections
# nginx      → Program name
# pid=1234   → Process number
# *:80       → Listening on all IPs on port 80
ℹ
Flags meaning : -l ( listening only ) , -n ( numeric — don't resolve names ) , -t ( TCP only ) , -p ( show the process ) .
2 . See all actual connections — Established

The difference from above : without -l to show connections that are actually happening .

ss -ntp | grep : PORT 
# Who is connected to port 80 right now?
ss -ntp | grep :80
# ESTAB  0  0  192.168.1.50:54000  192.168.1.10:80  users:(("nginx",pid=1234))

# Explanation:
# ESTAB           → Actual established connection
# 192.168.1.50    → Client IP that is connected
# 54000           → Client Port (random)
# 192.168.1.10    → Your server
# 80              → Server Port
3 . A Very Powerful Method — lsof

The lsof command gives you complete details about open files including network connections .

lsof -i : PORT

⚠

4 . LVM — The Complete Pipeline

LVM is the most popular storage method in servers . Instead of using the disk directly , you go through layers :

LVM Pipeline : PV → VG → LV 
Disk / Partition
    ↓
Physical Volume (PV)    ← pvcreate /dev/sdb
    ↓
Volume Group (VG)        ← vgcreate data_vg /dev/sdb
    ↓
Logical Volume (LV)      ← lvcreate -L 20G -n app_lv data_vg
    ↓
Filesystem               ← mkfs.xfs /dev/data_vg/app_lv
    ↓
Mount Point              ← mount /dev/data_vg/app_lv /data
Full Example — LVM from Scratch 
# 1. Physical Volume
pvcreate /dev/sdb
pvs                           # Confirm

# 2. Volume Group
vgcreate data_vg /dev/sdb
vgs                           # Confirm

# 3. Logical Volume (20G)
lvcreate -L 20G -n app_lv data_vg
lvs                           # Confirm

# 4. Filesystem
mkfs.xfs /dev/data_vg/app_lv

# 5. Mount
mkdir /data
mount /dev/data_vg/app_lv /data
df -h                         # Confirm
ℹ
Simple Analogy : PV = money in wallets , VG = bank account where you collect money , LV = amounts you distribute to projects . That's why LVM is powerful — you can combine different disks into one VG and distribute space the way you like .
5 . Create Filesystem

After finishing with LVM or Partition , you need to create a Filesystem to use the space . ( See Section 13 for the full comparison between XFS and Ext4 ) .

mkfs — Create Filesystem 
# XFS (best for servers — RHEL default)
mkfs.xfs /dev/data_vg/app_lv

# Ext4 (compatible — can be shrunk)
mkfs.ext4 /dev/data_vg/app_lv

# Or directly on a Partition
mkfs.xfs /dev/sdb1
⚠
Remember : XFS only supports Grow — it cannot be shrunk . Ext4 supports Grow and Shrink . See Section 13 for details .
6 . Mount and /etc/fstab

For users to use the space , you must mount it to a specific directory .

Mount and fstab 
# Create directory for mount
mkdir /data

# Direct mount
mount /dev/data_vg/app_lv /data

# Confirm
df -h
# Filesystem               Size  Used Avail Use% Mounted on
# /dev/mapper/data_vg-app_lv  20G  1G   19G   5% /data

# ── For persistence after reboot ──

# Find the UUID

# /dev/data_vg/app_lv: UUID="123-456" TYPE="xfs"

# UUID=123-456  /data  xfs  defaults  0  0

# Test before reboot
mount -a   # If no Error, it's good
Final Verification

df -h | grep /data

# /dev/mapper/data_vg-app_lv 20G 1G 19G 5% /data

The strongest feature of LVM — you can combine multiple disks of different sizes into one VG .

Troubleshooting — Diagnosis and Problem Solving

Troubleshooting is the skill of " systematic investigation " for the cause of a problem . Not just memorizing commands — it's a methodology : Define the problem , gather information , analyze , implement the solution , test .

Diagnostic Methodology — 5 Steps
  1. Define the Problem What exactly happened ? A user said " the site is down " or is there an alert from monitoring ?
  2. Gather Information logs , status , metrics ( dmesg , journalctl , top , df -h , ping ) . Start broad then narrow down .
  3. Analyze What's the pattern ? What changed before the problem ? ( last update , config change , traffic increase )
  4. Test the Hypothesis Apply a small fix . If it's a permissions issue ? Test with a single operation . If DNS ? Try nslookup from the server .
  5. Confirm the Solution and Document Is the problem solved ? Make sure it won't come back . Document the steps .
Common Problems and Solutions — Common Scenarios
1 . Server not responding ( SSH not working )

# Step 5: Check firewall rules

2 . Disk Full

HTTP 400 vs 500 — HTTP Errors

HTTP status codes are the server's way of telling the Client " what happened " . We divide them into two main categories : 4xx ( client error ) and 5xx ( server error ) .

4xx — Client Errors ( user error )
🔍
400 Bad Request
The request is not understood by the server — bad syntax , large header , payload doesn't match specifications .
🚫
401 Unauthorized
Not authorized — needs login . Authentication is missing or invalid .
⛔
403 Forbidden
The server understood the request but won't allow it — permissions , IP block , SELinux .
❓
404 Not Found
The resource doesn't exist . Most famous error — wrong path , file deleted , broken link .
⏱
408 Request Timeout
The client took too long to send the request . Usually due to slow internet or a proxy issue .
🔄
429 Too Many Requests
Rate limiting — the client exceeded the allowed request limit . Common in APIs .
5xx — Server Errors ( server error )
💥
500 Internal Server Error
General server error — no details . Usually due to application crash , PHP fatal error , or misconfiguration .
⚠
502 Bad Gateway
The server ( e.g. — Nginx ) tried to connect to upstream ( e.g. PHP-FPM ) and it didn't respond . Most common problem with reverse proxy .
🔌
503 Service Unavailable
The server can't serve — overload , maintenance , or connections exhausted .
⌛
504 Gateway Timeout
The upstream didn't respond in time . Usually due to a slow application or heavy database query .
Troubleshooting Guide — What to do when you see an error code
CodeFirst thing to check
400Check request headers and body — what exactly is the client sending
401Check authentication token or cookies
403Check permissions ( ls -la ) , SELinux ( ausearch -m avc ) , nginx.conf allow/deny rules
404Check path , root directive in nginx/Apache , symlinks
429Reduce rate from client or increase limits on server
500Check logs ( journalctl -u nginx , app logs , PHP error log )
502Make sure upstream ( PHP-FPM , uWSGI , Gunicorn ) is running : systemctl status php-fpm
503Is traffic high ? Check ss -s , top , connections limits

400 vs 500 — The Golden Rule
4xx ← problem is with the client ( user ) — fix the request .
5xx ← problem is with the server ( you ) — fix the service .

Rescue Mode & GRUB — System Rescue

Sometimes the system won't boot — GRUB is broken , the kernel won't start , or you forgot the password . Rescue Mode and GRUB Recovery are your tools to save the day .

Rescue Mode / Single User Mode

Rescue Mode gives you a shell as root without starting all services — to fix boot problems .

Entering Rescue Mode — from GRUB 
# During boot: press Shift (BIOS) or Esc (UEFI) to show GRUB
#
# 1. Highlight the first kernel option
# 2. Press 'e' to edit boot entry
# 3. Look for the line starting with linux or linux16
# 4. Add this word at the end of the line:
init=/bin/bash

# Or if systemd:
systemd.unit=rescue.target

# 5. Press Ctrl + X or F10 to boot
# 6. You'll get a shell as root — fix what you need

# After fixing — reboot
exec /sbin/init
# Or
reboot -f
⚠
Note : If you used init=/bin/bash , the system mounts as read-only . To write changes : mount -o remount,rw /
GRUB Recovery — Fixing the Bootloader

If GRUB itself is broken or corrupted ( e.g. after installing Windows ) , you need to reinstall it .

Fixing GRUB — from Live CD/USB 
# 1. Boot from Live CD/USB
# 2. Open terminal

# 3. Find the partition names (e.g. /dev/sda1 = /boot, /dev/sda2 = /)
lsblk

# 4. Mount the system root
mount /dev/sda2 /mnt
mount /dev/sda1 /mnt/boot   # If /boot is on a separate partition

# 5. Chroot — enter the system
for dir in /dev /proc /sys /run; do mount --bind $dir /mnt$dir; done
chroot /mnt

# 6. Reinstall GRUB
grub2-install /dev/sda         # Or grub-install depending on distribution

# 7. Rebuild GRUB config
grub2-mkconfig -o /boot/grub2/grub.cfg   # CentOS/RHEL/Fedora
update-grub                                 # Ubuntu/Debian

# 8. Exit and reboot
exit
reboot
ℹ
The Difference : grub-install writes GRUB to MBR or UEFI partition . grub-mkconfig ( or grub2-mkconfig ) builds the config file from the kernels installed in /boot .
Resetting root Password
Forgot root password — Reset it 
# Same steps as Rescue Mode:
# 1. From GRUB press 'e' on the kernel entry
# 2. Add init=/bin/bash at the end of the linux line
# 3. Ctrl + X to boot

# Remount as read-write
mount -o remount,rw /

# Change root password
passwd

# If SELinux is enabled
touch /.autorelabel

# Reboot
exec /sbin/init
GRUB Recovery — Quick Summary
Live CD → mount partitions → chroot → grub-install → grub-mkconfig → reboot
