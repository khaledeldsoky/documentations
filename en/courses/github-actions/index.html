GitHub Actions Guide
From Zero to CI/CD

GitHub Actions is a CI/CD system built into GitHub that lets you build, test, and deploy your code automatically . This guide is designed for beginners with no prior CI/CD or YAML experience . We explain every concept from scratch with practical examples .

CI/CD GitHub Actions YAML DevOps
1

What is GitHub Actions ?

GitHub Actions is a CI/CD ( Continuous Integration / Continuous Deployment ) system built into GitHub . Simply put : anything you do manually after a push — like running tests or deploying a site — can be automated by GitHub automatically .

❌
Before CI/CD — Manual
Push → open terminal → run tests → build → upload to server . Wastes time , you forget steps , errors happen .
✅
After CI/CD — Automated
Push → GitHub runs tests → builds → deploys automatically . You just focus on the code .
GitHub Actions Architecture
Workflow (the whole file — .github/workflows/*.yml)
Job 1 (e.g. build — runs on Ubuntu)
Step 1: actions/checkout (fetch the code)
Step 2: actions/setup-node (setup Node)
Step 3: run: npm ci (install deps)
Step 4: run: npm run build (build)
Job 2 (e.g. deploy — needs Job 1 to finish)
Step 1: download artifact (fetch the files)
Step 2: deploy to server (deploy)
Core Concepts
TermDescriptionExample
WorkflowYAML file that defines the entire automation.github/workflows/deploy.yml
JobGroup of steps running on the same runnerbuild , test , deploy
StepOne command or action inside a jobrun: npm test
ActionReusable unit from GitHub Marketplaceactions/checkout@v4
RunnerThe server that runs the workflowubuntu-latest , self-hosted
EventThe event that triggers the workflowpush , pull_request , schedule
ℹ
Let's agree on one thing : The Workflow is the whole file . The Job is a specific task inside the file . The Step is one line within the task . Don't memorize this — understand the relationship and it'll stick in your mind .
2

Your First Workflow

Let's start with the smallest possible workflow — it prints a message when you push . All you need is one file in the right place .

Where to Create the File ?
Mandatory Path 
# Create this folder in your repo root
mkdir -p .github/workflows/

# Then create .github/workflows/first.yml
touch .github/workflows/first.yml
⚠
Very important : The file must be inside .github/workflows/ exactly . If you put it anywhere else , GitHub won't read it . The extension must be .yml or .yaml .
Minimal Workflow
first.yml — Hello World 
# Workflow name (shows in GitHub)
name: Hello World

# When to run?
on:
  push:                      # any push on any branch

# Tasks
jobs:
  hello:                     # job name (any name)
    runs-on: ubuntu-latest # the server it runs on
    steps:
      - run: echo "Hello from GitHub Actions!"

After you push this file , go to GitHub → the Actions tab → you'll see the Workflow running . Click on it and check the logs .

How to See the Result ?
  1. Push the file Put the file in .github/workflows/first.yml and commit + push .
  2. Open GitHub ( your repo ) You'll find a tab called Actions at the top of the page — click it .
  3. Watch the Workflow run You'll find " Hello World " in the list — click it and watch the logs live .
  4. When you see a green check ✅ — congratulations ! Your first workflow is running ! It printed " Hello from GitHub Actions ! " in the log .
Summary

Workflow = YAML file in .github/workflows/ , triggered by an Event , runs Jobs with Steps on a Runner .

3

YAML Structure — Keyword Breakdown

Let's take each keyword in YAML and understand it individually . Every word has a specific role in the workflow . You don't need to memorize — understand the role and you'll know how to use it .

on — When to Run the Workflow ?

This is the first keyword in the workflow . It defines the event that triggers the file . It can be push , pull_request , schedule ( timer ) , or manual ( workflow_dispatch ) .

on Examples 
# on every push
on: push

# on push to main or dev only
on:
  push:
    branches: [main, dev]

# on pull_request to main
on:
  pull_request:
    branches: [main]

# schedule — daily at 12 midnight UTC
on:
  schedule:
    - cron: '0 0 * * *'

# manual — a button in GitHub
on:
  workflow_dispatch:
    inputs:
      env:
        description: 'Environment'
        required: true
        default: 'staging'
ℹ
Info : You can combine multiple events in the same workflow . For example, run on push and pull_request at the same time .
jobs — Tasks

Every workflow has one or more jobs . Each job runs on a separate runner ( independent server ) . You name the job whatever fits — no spaces .

jobs — basic 
jobs:
  build:           # ⬅️ any name you choose
    runs-on: ubuntu-latest
    steps: [ ... ]

  deploy:          # ⬅️ another job
    runs-on: ubuntu-latest
    steps: [ ... ]
PropertyDescriptionExample
runs-onRunner typeubuntu-latest , windows-latest
needsThe job that must finish firstneeds: build
ifCondition to runif: github.ref == 'refs/heads/main'
strategyRepeat the job with different valuesmatrix
envVariables for the whole jobenv: NODE_ENV: test
permissionsJob permissionspermissions: contents: read
steps — Steps

Every job has steps that run in order ( one after another ) . If a step fails , the rest won't run ( unless you set continue-on-error: true ) .

Step Types 
steps:
  # use — use a prebuilt action from GitHub
  - uses: actions/checkout@v4

  # run — direct shell command
  - name: Install dependencies
    run: npm ci

  # with — parameters for the action
  - uses: actions/setup-node@v4
    with:
      node-version: '20'

  # run with multi-line
  - name: Build and test
    run: |
      npm run build
      npm test

  # if — condition for the step
  - if: failure()
    run: echo "Something failed!"
runs-on — What does the Job run on ?

Every job needs a server to run on . You choose the type of server . GitHub-hosted are GitHub's servers , and Self-hosted are your own .

runs-on — options 
jobs:
  linux:
    runs-on: ubuntu-latest   # or ubuntu-22.04

  windows:
    runs-on: windows-latest # or windows-2022

  mac:
    runs-on: macos-latest   # or macos-14

  private:
    runs-on: self-hosted    # on your own machine
needs — Job Ordering

By default , all jobs run in parallel ( each job on its own runner at the same time ) . needs makes a job wait for another job to finish first .

needs — dependencies 
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building..."

  test:
    needs: build            # ⬅️ waits for build to finish
    runs-on: ubuntu-latest
    steps:
      - run: echo "Testing..."

  deploy:
    needs: [build, test] # ⬅️ waits for build and test
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying..."
💡
Flow : build ← test ← deploy . If build fails , test won't run at all .
strategy.matrix — Multi-run

The matrix lets you repeat the same job with different values ( e.g., test on 3 operating systems and 3 Node versions simultaneously ) .

Matrix — OS × Node 
jobs:
  test:
    strategy:
      fail-fast: false    # if one fails, the rest continue
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]

    runs-on: ${{ matrix.os }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci && npm test

This code produces 9 runs ( 3 OS × 3 Node versions ) — all running in parallel . You can add include or exclude to control the combinations .

env — Variables

You can define variables at 3 levels : for the entire workflow ( global ) , for a job , or for a single step .

env — 3 levels 
# Global — for the entire workflow
env:
  APP_NAME: my-app
  NODE_ENV: production

jobs:
  build:
    # Job level — for this job only
    env:
      BUILD_DIR: ./dist

    steps:
      - name: Build
        # Step level — for this step only
        env:
          API_URL: https://api.example.com
        run: echo $APP_NAME $BUILD_DIR $API_URL
permissions — Permissions

It defines the permissions of the GITHUB_TOKEN ( the token GitHub gives the workflow ) . Best practice is to reduce permissions as much as possible .

permissions — examples 
# Least privilege — read-only (best)
permissions:
  contents: read

# For a single job
jobs:
  deploy:
    permissions:
      contents: read
      pages: write      # for deploying to GitHub Pages
      id-token: write
if — Conditions

You use if to control a job or step — it only runs if the condition is met . You can use github variables ( like github.ref ) or functions like failure() and always() .

if — conditions 
# Job condition — only on main
jobs:
  deploy:
    if: github.ref == 'refs/heads/main'

    steps:
      # Step condition
      - if: failure()       # only if it failed
        run: echo "Failed!"

      - if: always()        # always — even on failure
        run: echo "Cleanup..."

      - if: success()       # on success — this is the default
        run: echo "All good!"
VariableExample ValueDescription
github.refrefs/heads/mainCurrent branch or tag
github.event_namepush , pull_requestThe event that triggered the workflow
github.actorkhaledThe user who triggered the event
github.repositoryowner/repoCurrent repo
github.shaabc123...The commit hash
defaults — Default Settings

It sets default settings for all run steps — like shell or working directory . Instead of repeating the same settings in every step .

defaults 
defaults:
  run:
    shell: bash
    working-directory: ./app

jobs:
  build:
    steps:
      - run: pwd  # will be in ./app
      - run: ls   # will be in ./app
Quick YAML Summary
Full Workflow Template 
name: Workflow Name     # workflow name

on: [push]                # when to run?

env:                      # global variables
  KEY: value

jobs:                     # tasks
  job-name:               # job name (any name)
    runs-on: ubuntu-latest # runner type
    needs: [job1]           # dependencies
    if: github.ref == '...' # condition
    permissions: {}          # permissions

    steps:                   # steps
      - uses: actions/checkout@v4
      - name: Description
        run: command
        env:                    # step variables
          KEY: value
        if: condition

From Theory to Practice

Let's apply what we've learned to a real scenario : a React site ( or Vue or any SPA ) we want to build and deploy to GitHub Pages — every time you push to main .

Scenario — 7 Steps
📖
Story : You're a developer with a React site . You write code , push , and want the site to deploy automatically without opening a terminal or clicking any buttons .
  1. You do : git push origin main GitHub receives the push and knows an event happened on main .
  2. GitHub looks for Workflow files Goes to .github/workflows/ and finds deploy.yml — starts reading it .
  3. GitHub provisions a Runner Prepares an ubuntu-latest server with Git and Node.js ready to go .
  4. Job 1 : Build Checkout ← Setup Node ← npm ci ← npm run build → produces a build/ folder .
  5. Upload Artifact Saves the build/ folder for the next job to use .
  6. Job 2 : Deploy ( needs : build ) Waits for build to finish ← downloads artifact ← deploys to GitHub Pages .
  7. Site LIVE ! 🚀 The new site is live at https://user.github.io/repo/ .
Complete Diagram
Job 1: BUILD
runs-on: ubuntu-latest
├─ actions/checkout@v4 ← "fetched the code ✅"
├─ actions/setup-node@v4 ← "setup Node.js 20 ✅"
├─ run: npm ci ← "installed dependencies ✅"
├─ run: npm run build ← "built → build/ ✅"
└─ upload-artifact ← "saved build/ ✅"
↓ needs : build " waiting for build to finish . . . " ↓
Job 2: DEPLOY
needs: build
if: github.ref == 'refs/heads/main'
├─ download-artifact ← "fetched build/ ✅"
└─ deploy-pages ← "deployed to Pages ✅"
← "Site LIVE! 🚀"
Full YAML
deploy.yml — React to GitHub Pages 
name: Build & Deploy React App

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: production-files
          path: ./build/

  deploy:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: production-files
          path: ./build/
      - uses: actions/deploy-pages@v4
        id: deployment
Comparison — Theory vs Practice
TheoryPractice ( YAML )
" run on push to main " on: push: branches: [main]
" on Ubuntu " runs-on: ubuntu-latest
" fetch the code " uses: actions/checkout@v4
" setup Node " uses: actions/setup-node@v4
" install deps " run: npm ci
" build the site " run: npm run build
" save the files " uses: actions/upload-artifact@v4
" wait for build to finish " needs: build
" fetch the files " uses: actions/download-artifact@v4
" deploy " uses: actions/deploy-pages@v4
Summary

You don't need to memorize YAML syntax . Understand the flow : Event → Job → Step . The syntax is just a way to describe that flow in a file .

GitHub-hosted vs Self-hosted

The Runner is the server that runs the workflow . Two types : GitHub-hosted ( provided by GitHub ) and Self-hosted ( you provide on your machine or server ) .

PropertyGitHub-hostedSelf-hosted
ServerGitHub's serversYour machine, VM, or server
Cost2000 minutes/month free ( Public repos ) Free — but server cost is on you
SetupReady — nothing neededNeed to download and run GitHub Actions Runner
Network accessPublic — can't reach private networkCan reach your internal network — VPN , VPC , etc .
MaintenanceGitHub is responsibleOn you — security , updates , availability
Use case99% of cases — sufficientWhen you need special hardware or a private network
Setting up a Self-hosted Runner
Installing Runner on Ubuntu 
# 1. Go to repo → Settings → Actions → Runners → Add runner
# 2. GitHub will give you commands like these:

mkdir actions-runner && cd actions-runner
curl -o actions-runner-linux-x64-2.xxx.tar.gz -L https://github.com/actions/runner/releases/download/v2.xxx/actions-runner-linux-x64-2.xxx.tar.gz
tar xzf actions-runner-linux-x64-2.xxx.tar.gz
./config.sh --url https://github.com/YOUR-ORG/YOUR-REPO --token ABC123
./run.sh
Using self-hosted in YAML 
jobs:
  deploy:
    runs-on: self-hosted   # runs on your machine
    steps:
      - run: echo "Running on my own server!"
⚠
Security warning : With a self-hosted runner, anyone with access to the repo can run code on your machine . Use with caution — especially in public repos . Better to use GitHub-hosted if you can .

Linking YAML Files

As the project grows , you'll need to reuse the same steps across multiple workflows . GitHub Actions supports two ways to link :

Method 1 : Reusable Workflows

A regular workflow but uses workflow_call instead of push — which lets other workflows call it .

reusable-build.yml — The Reusable Workflow 
name: Reusable Build

on:
  workflow_call:             # ⬅️ this is what makes it reusable
    inputs:
      node-version:
        required: true
        type: string

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm ci && npm run build
main.yml — The Call 
jobs:
  call-reusable:
    uses: ./.github/workflows/reusable-build.yml
    with:
      node-version: '20'
Method 2 : Composite Actions

A small reusable Action — it bundles a group of steps in an action.yml file . The difference is it's not a full workflow , just ready-made steps you use inside any job .

. github/actions/my-setup/action.yml 
name: My Setup
description: Setup Node and install deps

runs:
  using: composite
  steps:
    - uses: actions/setup-node@v4
      with:
        node-version: '20'
    - run: npm ci
      shell: bash
Using the composite action 
jobs:
  build:
    steps:
      - uses: actions/checkout@v4
      - uses: ./.github/actions/my-setup  # ⬅️ invocation
      - run: npm run build
💡
Difference : Reusable workflows = full workflows , Composite actions = ready-made steps for use inside any job .
Example — 3 Linked Files
.github/
├── workflows/
│ ├── main.yml # the main workflow
│ └── reusable-test.yml # reusable workflow
└── actions/
└── setup-node/ # composite action
└── action.yml
main.yml — calls reusable-test and composite action 
name: Main CI/CD
on: [push]

jobs:
  build:
    steps:
      - uses: actions/checkout@v4
      - uses: ./.github/actions/setup-node
      - run: npm run build

  call-tests:
    needs: build
    uses: ./.github/workflows/reusable-test.yml

  deploy:
    needs: [build, call-tests]
    steps:
      - run: echo "Deploying..."

Common Mistakes & Debugging

The most common mistakes when learning GitHub Actions — and how to fix them .

ProblemCauseSolution
Workflow not showing in GitHubFile is not in the right pathMake sure it's in .github/workflows/*.yml
" Invalid workflow file " Wrong YAML indentationUse two spaces ( not tabs )
" Command not found " Forgot actions/checkoutAdd - uses: actions/checkout@v4 as first step
" Permission denied " GITHUB_TOKEN doesn't have permissionsAdd permissions: in the YAML
" Secret not found " Secret not savedSettings → Secrets and variables → Actions
Build is slowNo cachingUse actions/cache@v4
Job stuck in queueGitHub concurrency limitsUse a self-hosted runner
Debugging Techniques
🔍
Enable Debug Logging
Add a secret named ACTIONS_STEP_DEBUG=true — GitHub will show detailed logs for every step .
⏩
continue-on-error
If a step fails , the whole workflow stops . Use continue-on-error : true to continue and see the rest .
⚠
if : failure ( )
Add a step at the end with if : always ( ) to see failure logs or do cleanup .
📋
Manual Dispatch
Use workflow_dispatch to run the workflow manually whenever you want for testing .
⚠
Most time-wasting mistake for beginners : YAML uses two spaces ( spaces ) not tabs . If you use a tab , GitHub will reject the file . Make your editor show whitespace .

Context Objects — Built-in Variables

GitHub Actions has Context Objects — built-in variables you can use in YAML to get info about the repo , the event , the runner , and more . You use them with ${{ context.key }} .

VariableDescriptionExamples
${{ github.ref }}Current branch or tagrefs/heads/main
${{ github.event_name }}The event that triggered the workflowpush , pull_request
${{ github.actor }}The user who triggered the eventkhaled
${{ github.repository }}The current repoowner/repo-name
${{ github.sha }}The commit hashabc123def456...
${{ env.VAR_NAME }}Variable from env${{ env.NODE_ENV }}
${{ secrets.SECRET_NAME }}Secret from Settings${{ secrets.API_KEY }}
${{ matrix.os }}Value from matrixubuntu-latest
${{ runner.os }}Runner OSLinux , Windows , macOS
Context Usage Examples 
jobs:
  info:
    steps:
      - run: echo "Branch: ${{ github.ref }}"
      - run: echo "Event: ${{ github.event_name }}"
      - run: echo "Actor: ${{ github.actor }}"
      - run: echo "OS: ${{ runner.os }}"
      - env:
          MY_SECRET: ${{ secrets.API_KEY }}
        run: echo "Key length: ${#MY_SECRET}"
⚠
Security warning : Don't print secrets in the logs ! If you write echo ${{ secrets.API_KEY }} , GitHub automatically masks it as *** . Still — don't print them .

Security in GitHub Actions

🔐
1 . Least Privilege
Always use permissions : contents : read — don't give more than the job needs .
🔑
2 . OIDC Instead of Secrets
OIDC lets the workflow get a temporary token from the cloud provider ( AWS , Azure , GCP ) without storing a long-lived secret .
🛡️
3 . Avoid Script Injection
Don't use ${{ }} directly inside run: — a user could inject malicious code via a PR title for example . Use env: to pass the values .
📌
4 . Pin Actions with SHA
Instead of @v4 , use @full-sha-hash to prevent unexpected changes in the action from affecting the workflow .
Script Injection — The Problem and Solution 
# ❌ Wrong — vulnerable
- run: echo "${{ github.event.pull_request.title }}"

# ✅ Right — safe
- env:
    PR_TITLE: ${{ github.event.pull_request.title }}
  run: echo "$PR_TITLE"
ℹ
Security summary : Least privilege → OIDC if possible → don't trust external input → pin action versions .

Interview Questions

The most common questions in GitHub Actions interviews — with ideal answers .

Basic Questions
Q
Explain GitHub Actions briefly .
• It's a CI/CD system from GitHub . You write YAML files in .github/workflows/ that define Events ( like push ) that trigger Jobs ( tasks ) with Steps ( steps ) on Runners ( servers ) .
Q
What's the difference between push and pull_request events ?
•push runs when you push to a branch . pull_request runs when you create a PR ( open , synchronize , closed ) .
Q
What's the difference between needs and strategy.matrix ?
•needs makes a job wait for another job ( sequential execution ) . •matrix lets you repeat the same job with different values ( parallel execution ) .
Q
How do you deploy after tests pass ?
•Use needs: test in the deploy job — it won't run unless test succeeds . And also if: github.ref == 'refs/heads/main' for main only .
Practical Scenarios in Interviews
Q
You have a workflow that runs on push to main , and you want a manual trigger to choose the environment .
•Use workflow_dispatch with an input for the environment .
Manual Trigger 
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment to deploy'
        required: true
        default: 'staging'
Q
How do you speed up the workflow with caching ?
•Use actions/cache@v4 to save node_modules or ~/.npm and restore them next time instead of downloading from scratch .
Caching node_modules 
- name: Cache node_modules
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-node-
🎯
Interview tip : Start with the basics ( Workflow ← Job ← Step ) . Mention actions/checkout and actions/setup-node as examples . Talk about caching and secrets as best practices . If asked about scale : mention reusable workflows and self-hosted runners .
